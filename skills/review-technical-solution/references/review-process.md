# 技术方案评审流程补充示例

这个文档只提供补充示例，不新增主路径规则。执行顺序、停机条件、严重级别和输出字段要求一律以 `skills/review-technical-solution/SKILL.md` 为准。

## 1. 适合查这个文档的场景

- 主技能已经足够执行，但你想看主张提取的字段展开示例。
- 你想核对代码取证三状态的判断方式。
- 你想快速自检输出前的检查清单。

## 2. 主张提取示例骨架

```markdown
- `claim_id`：CLAIM-01
- `statement`：[方案声称什么]
- `assumption`：[依赖什么前提]
- `expected_evidence`：[需要在哪些代码/配置/接口中验证]
- `risk_if_wrong`：[如果不成立会怎样]
```

补充提醒：

- `statement` 要具体到能直接对应代码或配置，不要写成泛泛的讨论议题。
- `expected_evidence` 要明确文件、接口或表名，方便后续直接定位。
- `risk_if_wrong` 要能直接对应严重级别判断。

## 3. 代码取证三状态说明

- `已证实`：代码或配置中存在直接证据，足以支持该主张。
- `已证伪`：代码现状与主张冲突，或明确不存在方案声称的能力、边界或依赖。
- `待核验`：当前看不到足够证据，或需要额外代码入口、运行时信息、外部系统上下文才能确认。

补充提醒：

- 不能用"应该有""大概率有"处理缺失证据，必须写 `已证伪` 或 `待核验`。
- 如果只找到部分证据，要明确哪些部分 `已证实`，哪些部分仍 `待核验`。

## 4. 评审维度归因说明

- `需求对齐`：方案是否真正覆盖需求目标、边界、非目标和约束。
- `架构对齐`：方案是否符合 `.architecture/principles.md` 中的原则和边界要求。
- `代码现状对齐`：方案对现有模块、依赖、接口、数据结构和运行方式的判断是否符合代码现实。
- `完整性`：是否遗漏关键流程、失败路径、迁移、回滚、监控、测试或发布安排。
- `可落地性`：实现路径、依赖前提、资源成本、协作条件和验证方式是否足够具体、可执行。

补充提醒：

- 一个问题可以同时命中多个维度，保留多维归因，不强行只选一个标签。

## 5. 改进建议补充说明

改进建议要写清：建议动作、涉及对象、预期修正结果、完成后应补的验证证据。

补充提醒：

- 如果问题来自信息缺失，改进方案应明确要补哪类资料、代码入口或测试证据。
- 改进建议优先写可执行动作，不写空泛口号。

## 6. 输出前自检清单示例

- [ ] 四类必要输入是否齐全且本次确实已查看
- [ ] 每条关键结论是否都能回溯到需求、方案、原则或代码证据
- [ ] 是否把猜测和事实严格区分，未证实内容是否标为 `待核验`
- [ ] 复用声明是否都已在代码中找到证据或被明确指出缺失
- [ ] 问题分级、最终结论、改进建议是否彼此一致
- [ ] 输出顺序是否仍符合固定六区块结构
