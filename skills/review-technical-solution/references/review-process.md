# 技术方案评审流程参考

这个文档只定义“怎么评审”。它负责输入校验、主张提取、代码取证、问题分级和输出前自检，不替代 `skills/review-technical-solution/SKILL.md` 的主入口职责。

## 1. 文档定位

- 主 skill 负责顶层执行边界、停机规则、固定输出顺序和红旗信号。
- 本文补充正式评审时的细化流程，确保每条结论都有来源、有证据、有分级。
- 本文不负责按主题生成方案，也不负责替代需求、方案正文或项目原则。

## 2. 输入校验

正式评审前必须确认以下四类输入都存在且可读：

- `需求详情`
- `技术方案文档`
- `.architecture/principles.md`
- `相关项目代码`

任一输入缺失、不可读、范围不清或与当前评审对象不匹配时，直接输出 `无法开展正式评审`。

输出 `无法开展正式评审` 时必须同时说明：

- 缺少或不可用的具体输入是什么
- 为什么这会阻断正式评审
- 需要补充什么信息或代码入口后才能继续

## 3. 方案分类与评审焦点确定

- 先根据方案目标判断主类型，再判断是否同时命中多个类型。
- 分类依据使用 `references/review-analysis-guide.md`。
- 如果同时命中多类：评审问题取并集，必查证据取并集，最终结论按最高严重级别判定。
- 分类只是帮助确定重点，不得因为某一类检查通过就跳过其余必要核验。

## 4. 主张提取

从 `需求详情` 与 `技术方案文档` 中提取需要被代码或事实验证的主张。至少提取以下信息：

- `claim_id`
- `statement`
- `assumption`
- `expected_evidence`
- `risk_if_wrong`

建议把每条主张写成“方案声称什么 / 依赖什么 / 需要在哪里验证 / 如果不成立会怎样”的形式。

重点提取以下主张：

- 复用现有能力的声明
- 新增模块、接口、表结构、配置或流程的声明
- 兼容性、迁移、回滚、灰度或发布策略的声明
- 安全、性能、容量、成本、合规等非功能性声明

## 5. 代码取证

对每条主张在 `相关项目代码` 中寻找直接证据。证据可以来自实现、配置、schema、接口定义、测试、迁移脚本或明确的调用关系。

每条主张只能落在以下三种状态之一：

- `已证实`：代码或配置中存在直接证据，足以支持该主张。
- `已证伪`：代码现状与主张冲突，或明确不存在方案声称的能力、边界或依赖。
- `待核验`：当前看不到足够证据，或需要额外代码入口、运行时信息、外部系统上下文才能确认。

硬性规则：

- 方案声称“复用已有能力”时，必须在代码中找到对应实现、接口、配置或测试证据。
- 找不到复用能力时，不能按“应该有”处理；必须标记为 `待核验` 或 `已证伪`。
- 不能用经验、常识或方案自述替代代码证据。
- 如果只找到部分证据，要明确哪些部分 `已证实`，哪些部分仍 `待核验`。

## 6. 评审维度判定

所有问题都要归因到以下至少一个维度：

- `需求对齐`：方案是否真正覆盖需求目标、边界、非目标和约束。
- `架构对齐`：方案是否符合 `.architecture/principles.md` 中的原则和边界要求。
- `代码现状对齐`：方案对现有模块、依赖、接口、数据结构和运行方式的判断是否符合代码现实。
- `完整性`：是否遗漏关键流程、失败路径、迁移、回滚、监控、测试或发布安排。
- `可落地性`：实现路径、依赖前提、资源成本、协作条件和验证方式是否足够具体、可执行。

如果一个问题同时命中多个维度，保留多维归因，不强行只选一个标签。

## 7. 问题分级规则

- `blocker`：结论成立会直接导致方案不能安全推进，例如关键主张被代码证伪、必要依赖不存在、违反核心原则、缺失不可替代的落地路径。
- `major`：不会立刻阻断讨论，但若不修改，高概率导致实现返工、严重风险外溢、关键目标落空或上线失败。
- `minor`：问题真实存在，但影响范围有限，可在进入实现前一并修正，不足以单独改变总体方向。
- `note`：提醒、观察或建议项，不构成缺陷结论，但值得在文档或实现阶段关注。

分级必须基于证据和影响，不得因为用户紧急程度、作者资深程度或方案文字完整度而调整等级。

## 8. 最终结论规则

- 任何 `blocker` -> `阻断`
- 缺少必要输入 -> `无法开展正式评审`
- 无 `blocker` 但存在一个或多个 `major` -> `需修改`
- 仅有 `minor` / `note` 且核心主张已被充分证实 -> `通过`
- 如果仍存在关键 `待核验` 主张且其风险足以影响方向判断，应提升为 `需修改` 或 `阻断`，不能当作 `通过`

最终结论必须与问题列表、分级和证据状态一致，不能只给抽象态度。

## 9. 改进建议规则

- 每个 `blocker` 和 `major` 都必须给出明确改进方案。
- 改进方案要写清：建议动作、涉及对象、预期修正结果、完成后应补的验证证据。
- 如果问题来自信息缺失，改进方案应明确要补哪类资料、代码入口或测试证据。
- 改进建议优先写可执行动作，不写空泛口号。

## 10. 输出前自检

正式输出前逐项确认：

- 四类必要输入是否齐全且本次确实已查看
- 每条关键结论是否都能回溯到需求、方案、原则或代码证据
- 是否把猜测和事实严格区分，未证实内容是否标为 `待核验`
- 复用声明是否都已在代码中找到证据或被明确指出缺失
- 问题分级、最终结论、改进建议是否彼此一致
- 输出顺序是否仍符合主 skill 的固定结构
