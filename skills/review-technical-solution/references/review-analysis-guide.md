# 技术方案评审分类指引

这个文档只定义“按什么类型看方案”。它负责分类判定、各类方案的必查问题、必查代码证据和常见阻断项，不替代主 skill 的停机规则与固定输出。

## 1. 文档定位

- 主 skill 负责正式评审的入口、停机规则和输出结构。
- `references/review-process.md` 负责通用评审流程。
- 本文负责帮助评审者识别方案类型，并据此补足必须关注的问题和证据。

## 2. 分类判定原则

- 先判断方案的主要目标是什么，再判断是否同时覆盖多个目标。
- 分类依据看“核心变更意图”，不是只看标题关键词。
- 当一个方案同时命中多类时：评审问题取并集，必查证据取并集，最终结论按最高严重级别判定。
- 如果分类拿不准，先按更高风险的类别补查，不得因为分类模糊而缩小评审范围。

## 3. 五类方案

1. 新功能方案
2. 重构或替换方案
3. 跨系统或平台能力方案
4. 安全 / 合规 / 数据保护方案
5. 性能 / 容量 / 成本优化方案

当一个方案同时命中多类时：评审问题取并集，必查证据取并集，最终结论按最高严重级别判定。

## 4. 各类重点

### 1. 新功能方案

必查问题：

- 需求目标、非目标和影响范围是否清楚
- 新能力落在哪些模块、接口、数据结构和调用链上
- 是否说明失败路径、兼容策略、测试和发布安排

必查代码证据：

- 现有可复用模块、接口、schema 或基础能力是否真实存在
- 与目标功能相邻的现有边界、调用入口和数据模型是什么
- 是否已有类似实现、测试模式或约束需要沿用

常见阻断项：

- 方案依赖的现有能力在代码中不存在
- 新功能边界与现有模块职责直接冲突
- 只有主流程，没有失败路径、兼容或验证设计

### 2. 重构或替换方案

必查问题：

- 当前痛点是否有代码现实或指标支撑
- 替换边界、兼容要求、迁移步骤和回滚路径是否清楚
- 是否把无关优化混入同一次重构

必查代码证据：

- 现有实现的真实依赖、入口、调用链和耦合点
- 旧实现是否存在难以替换的隐藏约束、共享状态或运行时假设
- 迁移所需的测试、脚本、适配层或兼容层现状

常见阻断项：

- 没有可执行的迁移或回滚路径
- 低估现有依赖复杂度，方案与代码现状不符
- 重构范围失控，超出可管理边界

### 3. 跨系统或平台能力方案

必查问题：

- 参与系统、平台、团队和职责边界是否明确
- 接口、事件、同步 / 异步链路、失败补偿和恢复策略是否完整
- 是否说明跨边界依赖的拥有者、变更节奏和协作前提

必查代码证据：

- 系统间接口定义、事件契约、网关、任务编排或同步机制是否存在
- 相关代码中当前的边界、依赖方向、重试或补偿机制是什么
- 是否已有共享平台能力、SDK、适配层或中间件可复用

常见阻断项：

- 关键跨系统依赖在代码或接口契约中不存在
- 只设计成功链路，没有失败恢复与补偿策略
- 职责边界模糊，无法判断谁负责落地和运维

### 4. 安全 / 合规 / 数据保护方案

必查问题：

- 保护对象、威胁面、权限边界和合规约束是否说清
- 认证、授权、审计、脱敏、保留周期和删除策略是否落到具体流程
- 安全事件检测、追踪、隔离和恢复路径是否具备

必查代码证据：

- 认证鉴权、审计日志、密钥管理、敏感字段处理和数据访问控制的现状
- 相关接口、存储、日志和任务中是否已有合规约束或例外处理
- 测试、配置或中间件中是否已有可复用的安全控制能力

常见阻断项：

- 把安全 / 合规要求停留在文字层，没有代码或流程落点
- 缺少审计、追踪或敏感数据保护设计
- 方案与现有权限模型或数据处理方式直接冲突

### 5. 性能 / 容量 / 成本优化方案

必查问题：

- 当前瓶颈、容量压力或成本问题是否有基线数据支撑
- 优化目标值、测量方法、降级策略和副作用是否明确
- 是否说明对一致性、复杂度、可维护性和成本结构的影响

必查代码证据：

- 当前热点路径、资源消耗点、缓存策略、批处理方式或扩容机制的真实实现
- 现有监控、压测、容量预估或成本归因手段是否存在
- 方案依赖的优化能力是否已在代码或基础设施中具备

常见阻断项：

- 没有基线数据就下结论优化
- 方案依赖的性能能力或扩容机制并不存在
- 只追求局部指标，缺少全局复杂度与风险控制
