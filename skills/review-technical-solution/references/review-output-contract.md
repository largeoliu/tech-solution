# 技术方案评审输出契约

## 1. 文档定位

- 本文只定义正式评审输出必须长什么样。
- 主入口负责什么时候评，`references/review-process.md` 负责怎么评，本文负责最后怎么写。
- 所有区块都必须出现；如果某个区块没有内容，必须写 `- 无`，不能省略标题。

## 2. 固定输出顺序

正式评审输出必须严格按以下顺序展开：

1. `评审结论`
2. `阻断项`
3. `主要问题`
4. `改进方案`
5. `待补充信息`
6. `建议验证`

规则：

- 不得调换顺序。
- 不得合并区块。
- 不得新增位于这六个区块之间的自定义标题。
- 任一区块为空时，写 `- 无`。

## 3. 允许的结论值

`评审结论` 只能填写以下四个值之一：

- `通过`
- `需修改`
- `阻断`
- `无法开展正式评审`

除上述四个值外，不得输出其他同义结论。

## 4. 固定字段

### 4.1 `评审结论`

- `conclusion`
- `summary`

### 4.2 每个问题项（适用于 `阻断项` 与 `主要问题`）

- `severity`
- `category`
- `problem`
- `evidence`
- `impact`
- `recommendation`
- `validation`
- `targets`

### 4.3 每个改进项（适用于 `改进方案`）

- `category`
- `recommendation`
- `validation`
- `targets`

### 4.4 `待补充信息`

- `missing_information`
- `reason`
- `targets`

### 4.5 `建议验证`

- `validation`
- `targets`

## 5. 证据规则

- `evidence` 必须引用需求详情、技术方案文档、`.architecture/principles.md`、项目代码、配置、schema、接口定义或测试中的具体证据。
- 未证实内容不得写成事实；证据不足时，应在问题或说明中明确仍需核验。
- `targets` 必须指向建议关注的模块、文件、接口、表、任务、配置项或测试入口，不能只写空泛对象名。

## 6. 严重级别联动规则

- 出现任一 `severity: blocker` 时，`评审结论` 不能写 `通过`。
- 存在一个或多个高风险未解决问题时，`评审结论` 应与最高严重级别保持一致。
- 如果因缺少必要输入而无法完成正式评审，直接输出 `无法开展正式评审`，并在 `待补充信息` 中说明缺口。

## 7. 禁止输出模式

- 所有区块都必须出现。
- 不得省略固定区块。
- 不得输出没有 `evidence` 支撑的问题结论。
- 不得把 `targets` 留空后仍声称建议可执行。
- 不得用散文式段落替代固定字段清单。
