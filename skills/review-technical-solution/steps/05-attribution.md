# 步骤 5：归因与分级

## 输入
- 代码取证结果（step-4 产物）
- 方案分类（step-2 产物）
- `.architecture/principles.md`
- 状态文件（含 step-4 结果）

## 操作
1. 所有问题归因到五维度之一：需求对齐、架构对齐、代码现状对齐、完整性、可落地性
2. 以下模式默认归入 `代码现状对齐`：未盘点现有资产即提出新建、已存在低风险改造路径却仍平行建设、只用空泛理由回避复用/改造比较
3. 按严重级别分级：blocker/major/minor/note
4. 分级基于证据和影响，不因人调整
5. 分级下限：未比较现有候选就通过新增、拆分、合并、迁移、平行建设、职责转移或能力增强制造重复能力，至少 `major`；已有明确低风险复用/改造路径却仍制造重复能力，记 `blocker`；已比较候选但证据不足，至少 `major`；已比较候选但局部论证不完整，记 `minor`
6. `低风险复用/改造` 至少同时满足：职责边界无需重写、兼容性影响可控、主要调用方可保持兼容或低成本迁移

## 完成标准
- 所有问题都已补齐归因维度与严重级别
