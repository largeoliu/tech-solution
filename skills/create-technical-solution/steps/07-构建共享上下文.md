# 步骤 7：构建共享上下文

## 输入
- `.architecture/principles.md`
- 代码与配置
- 现有实现
- 相关文档
- repowiki（如存在）
- 步骤 3 记录的模板结构快照
- 状态文件

## 操作
1. 整合原则、现状、已有实现、关键约束与模板结构快照，形成共享上下文
2. 凡槽位涉及新增、拆分、合并、迁移、平行建设、职责转移或能力增强，必须建立“现有资产候选”上下文条目；每个受影响能力单元必须单独成条，不得合并成泛化清单。每条候选至少包含：`资产类型`、`资产标识`、`位置`、`当前职责`、`当前能力`、`可扩展点`、`已知限制`、`调用方/依赖方`、`相关证据路径`、`适用槽位`
3. 每条上下文包含：上下文编号（CTX-01 递增）、来源、结论或约束、适用槽位、可信度或缺口；其中 `来源` 必须可追溯到具体文件、目录、配置项或用户输入，`可信度或缺口` 至少区分 `已验证` / `待补证` / `存在冲突`
4. 搜索现有候选时，搜索范围至少覆盖：同职责对象、命名相近对象、共享基础层、直接入口或调度层
5. 若搜索后仍未发现现有候选，必须把搜索范围、搜索关键词、已排除目录或对象、未发现结论写入 CTX，不得以“暂未看到”跳过
6. 若某条上下文无法映射到任何槽位，必须记录原因；若关键槽位缺少可用上下文，必须标记阻塞，不得带隐含假设继续
7. 若 repowiki 存在，先读取目录、导航与总览，再补充与当前方案直接相关的内容，并记录已覆盖范围与仍待补证范围
8. 将 WD-CTX 区块写入 working draft
9. 向用户展示摘要

## 完成标准
- 每条上下文都有完整字段，且来源可追溯
- 无法映射到槽位的上下文已记录原因
- `WD-CTX` 已覆盖原则、现状、已有实现、关键约束、当前模板结构与现有资产候选；缺失项已明确标记为 `待补证` 或 `存在冲突`
- 关键槽位上下文已齐全或识别为阻塞项，不存在未显式记录的隐含假设
- 涉及新增、拆分、合并、迁移、平行建设、职责转移或能力增强的槽位，已按能力单元逐条建立现有资产候选清单，或已逐条记录“未发现候选”的搜索证据
- 如存在 repowiki，已记录相关范围覆盖情况或待补证范围
- WD-CTX 区块已写入 working draft
- 摘要已向用户展示

## 输出
- 更新状态文件 checkpoints.step-7
- WD-CTX 写入 working draft

## 门控
关键槽位无 CTX 支撑，存在无法追溯来源的 CTX 项，或涉及新增、拆分、合并、迁移、平行建设、职责转移或能力增强的槽位缺少按能力单元逐条建立的现有资产候选清单（含未发现候选的逐条搜索证据）时停止，不得进入第 8 步。

## 回退信号
共享事实、原则解释、现状判断、关键约束变化。
