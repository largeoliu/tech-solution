# 步骤 11：严格模板成稿并保存结果

## 输入
- 当前生效模板
- 协作收敛纪要
- working draft
- 状态文件

## 操作
1. 进入本步骤前必须先核验状态文件门禁：运行 `python scripts/validate-state.py --state <状态文件路径> --step 11 --flow-tier <flow_tier> --format json` 验证门禁。若未通过，优先根据 `repair_plan[]` 修复，并遵循其中 `depends_on_steps` 的前置顺序，且按 `completion_checks` 确认每项修复已完成；若还需补充判断，再参考 `summary.recommended_repair_sequence`、`summary.recommended_rollback_step`、`summary.missing_artifacts` 与 `issues[*].repair_guidance` 后重试；通过后再生成最终文档
2. 以模板为底稿，从 SLOT-01 开始逐槽填充
3. 每个槽位只能写入 `WD-SYN` 或 `WD-SYN-LIGHT` 中该槽位的收敛结论
4. 凡槽位涉及新增、拆分、合并、迁移、平行建设、职责转移或能力增强，成稿时必须按收敛区块中的 `建议落位槽位` 写入当前模板已有槽位，内容包括：
   - 现有候选比较
   - 选定路径
   - 支撑该路径成立的关键证据
   若选择 `新建`，相关不可复用、不可改造、新建必要性证据必须已经体现在 `关键证据引用` 中
   不得新增模板外章节、固定表格、固定列名或隐含小节
5. 若某项必要结论在不改变当前模板结构的前提下无法承载，必须记录为 `模板承载缺口` 并阻塞，不得擅自扩展模板
6. 无收敛内容时保留占位说明
7. 创建缺失目录（`.architecture/technical-solutions/working-drafts/` 或 `.architecture/technical-solutions/`）
8. 写入最终文档到 `.architecture/technical-solutions/[slug].md`
9. 如文件已存在，默认追加时间戳后缀另存，并在 `checkpoints.step-11` 中记录最终落盘路径；仅在用户明确要求覆盖时才覆盖原文件
10. 成稿成功后，方可把状态文件 `can_enter_step_12` 置为 true
11. 若成稿过程中发生推理失败、写入失败、标题顺序校验失败或模板快照不一致：
   - 必须停留在步骤 11
   - 保留 working draft 和状态文件
   - 重新运行 `validate-state.py --step 11/12 --format json`
   - 优先根据 `repair_plan[]` 回退步骤 3 或重做步骤 11
   - 禁止直接把 `current_step` 推到 12

## 完成标准
- 一级章节数量和顺序与模板完全一致
- 所有已收敛内容都找到语义归宿
- 所有受影响能力都已按 `建议落位槽位` 写入当前模板已有槽位：
  - 带有支撑选定路径成立的关键证据
  - 所有 `新建` 结论的关键证据已覆盖不可复用、不可改造与新建必要性说明
- 不存在通过新增模板结构来承载治理结论的情况
- 不存在未解决的 `模板承载缺口`
- 无回退到默认模板痕迹
- 文件已写入目标目录
- 如目标文件已存在，已按默认另存策略处理或按用户明确要求覆盖

## 输出
- 更新状态文件 checkpoints.step-11
- 最终文档写入 `.architecture/technical-solutions/[slug].md`

## 门控
以下情况不能执行吸收检查，不得进入第 12 步：
- 最终成稿未完成
- 任一受影响能力缺少现有候选比较 / 选定路径 / 落位结果
- 任一 `新建` 结论的 `关键证据引用` 未覆盖不可复用 / 不可改造 / 新建必要性说明
- 仍存在未解决的 `模板承载缺口`

## 回退信号
只影响最终模板落位。
