# 步骤 3：读取当前生效模板

## 输入
- `.architecture/templates/technical-solution-template.md`
- 状态文件

## 操作
1. **【强制】前置条件自检**：
   - 运行 `python scripts/validate-state.py --state <状态文件路径> --step 3 --flow-tier <flow_tier> --format json`
   - 确认步骤 2 已完成（prerequisites_checked 为 true）
   - 若验证失败，按 `repair_plan[]` 修复后重试
   - 展示自检结果（通过/不通过 + 具体原因）
2. 读取模板标题、章节层级、说明文字、占位说明；不得由模型自行概括模板结构
3. 沿用步骤 1 的文件名（slug），但 slug 不再单独保存在 state；后续仅通过 `working_draft_path` / `final_document_path` 体现
4. 确定 `solution_root`：兼容读取历史 `.architecture/solutions/`，但本次流程的新产物统一写入 `.architecture/technical-solutions/`
5. 预留 working draft 路径：`.architecture/technical-solutions/working-drafts/[slug].working.md`
6. 运行 `python scripts/extract-template-snapshot.py --template .architecture/templates/technical-solution-template.md --slug <slug> --working-draft .architecture/technical-solutions/working-drafts/<slug>.working.md --state <状态文件路径> --write`
7. 只接受脚本生成的模板指纹与槽位数量；state 不再保存完整 `template_snapshot` 正文
8. 将模板元信息与槽位清单写入 working draft 头部
9. 将 `checkpoints.step-3.template_loaded: true`、`checkpoints.step-3.template_fingerprint`、`checkpoints.step-3.slot_count`、`solution_root`、`working_draft_path` 写入状态文件
10. 展示模板结构摘要并自动继续

## 完成标准
- 模板标题、一级章节数量和顺序已记录
- 每个一级章节语义已明确
- working draft 路径已创建并写入模板结构快照
- 模板结构摘要已展示
- validator 对步骤 3 的即时复核通过，即 `template_fingerprint` 与当前模板完全一致

## 输出
- 更新状态文件 checkpoints.step-3
- working draft 头部写入模板结构快照

## 门控
模板结构摘要未稳定时不能判断方案类型，不得进入第 4 步。

## 回退信号
当前生效模板发生变化。
