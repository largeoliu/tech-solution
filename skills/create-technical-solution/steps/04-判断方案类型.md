# 步骤 4：判断方案类型

## 输入
- 方案主题
- 模板结构
- 状态文件

## 操作
1. **【强制】前置条件自检**：
   - 运行 `python /path/to/run-step.py --state <状态文件路径>` 查看当前门禁与修复建议
   - 确认步骤 3 已完成（template_loaded 为 true、working_draft_path 已创建）
   - 若验证失败，按 `repair_plan[]` 修复后重试
   - 展示自检结果（通过/不通过 + 具体原因）
2. 按"方案类型速判"判断命中哪一类或哪几类（新功能方案、重构或替换方案、跨系统或平台能力方案、安全/合规/数据保护方案，性能/容量/成本优化方案）
3. 确定必答问题、易漏风险、评审重点
4. 确定推荐参与成员
5. 所有流程统一为 full 级别，中间产物固定为 `WD-CTX`、`WD-TASK`、`WD-EXP-SLOT-*`、`WD-SYN-SLOT-*`
6. 标准入口是先运行 `python /path/to/run-step.py --state <状态文件路径> --advance` 进入本步骤；脚本会返回 `business_task` 与 `required_output_shape`。提交本步业务结论时，使用 JSON 正文原子写入：`python /path/to/run-step.py --state <状态文件路径> --complete --ticket <ticket> --summary "<步骤4完成>" <<'HEREDOC'`
   `{"solution_type": "<方案类型>"}`
   `HEREDOC`
7. 提交后写入：
    - `checkpoints.step-4.solution_type`
    - `required_artifacts`（固定为 `["WD-CTX", "WD-TASK", "WD-EXP-SLOT-*", "WD-SYN-SLOT-*"]`）
    严禁先推进步骤，再手工补改状态文件

## 职责边界

本步骤只做轻量类型分类，输出一个 `solution_type` 值。以下工作属于后续步骤，不得在本步提前执行：

| 禁止行为 | 正确归属步骤 |
|----------|-------------|
| 大规模搜索或阅读业务实现代码 | 步骤 7（构建共享上下文） |
| 逐能力单元建立现有资产候选清单 | 步骤 7（构建共享上下文） |
| 对复用 / 改造 / 新建做逐条路径比较 | 步骤 9（专家分析）和步骤 10（协作收敛） |
| 为判断方案类型而深度理解领域模型 | 步骤 7（构建共享上下文） |

判断方案类型只需基于：需求文档、步骤 1 的定题与范围、模板结构和项目原则。不需要先完成代码考古才能给出类型判断。

## 完成标准
- 方案主类型已确定
- 参与成员集合已初步确定
- 必答问题、易漏风险、评审重点已记录

## 输出
- 更新状态文件 checkpoints.step-4

## 门控
方案类型未确定时不能进入第 5 步。

## 回退信号
方案类型判断本身发生变化。
