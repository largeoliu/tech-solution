# 步骤 9：组织专家按模板逐槽位分析

## 输入
- 共享上下文
- 模板任务单
- repowiki 相关内容（如存在）
- working draft
- 状态文件

## 操作
1. 进入本步骤前先核验状态文件门禁：`produced_artifacts` 已包含 `WD-CTX`、`WD-TASK`，且 `can_enter_step_9` 为 true

**checkpoint模板**：`step-9.summary = "完成；写入专家分析；slots={N}；gate: step-10 ready"`

 2. 改为按模板槽位组织分析，而不是让每位专家完整覆盖全部槽位。逐槽位组织分析（每次分析一个槽位），但所有槽位的专家分析必须在**单次** `--complete` 提交流程中以完整 JSON payload 一次性提交，不得拆分成多次调用
3. 同一槽位内，相关专家仍按顺序给出独立判断，但只需读取“当前槽位相关的前序摘要”与已写入的该槽位 `WD-EXP-SLOT-*` 片段；不得要求其读取无关槽位的历史产物
4. 若步骤 6 确认存在 repowiki，相关槽位分析必须纳入其中与该槽位直接相关的内容；repowiki 只能补充判断，不得替代 CTX 编号 grounding
5. 每位专家只对自己参与的槽位写判断，不得提前收敛为统一口径；每个参与槽位至少写出：`参与槽位`、`决策类型`（`复用` / `改造` / `新建`）、`核心理由`、`关键证据引用`、`未决点`
6. 若 `决策类型` 为 `新建`，必须显式回答“为什么不能复用现有对象”和“为什么不能在现有对象上改造”，且理由必须绑定 CTX 编号，不得只写风格偏好
7. 先运行 `python /path/to/run-step.py --state <状态文件路径> --advance` 进入本步。脚本会自动处理 step card、门禁和 ticket，并返回本步 `business_task`、`artifact`、`required_output_shape`
8. 每位专家完成当前槽位分析后，只提交 **结构化 JSON payload**；脚本会按槽位把 payload 渲染成 `WD-EXP-SLOT-*` 文件，并同步 `produced_artifacts`
9. 当前步骤的 ticket 只允许写入 `WD-EXP-SLOT-*`；若 `--advance` 发 ticket 后 state、working draft 或 block 范围变化，必须重新执行 `--advance`
10. 当所有需要专家判断的槽位都已拥有足够的 `WD-EXP-SLOT-*` 片段后，才把状态文件 `can_enter_step_10` 置为 true
11. state 中不得保存任何专家分析正文；专家判断只允许写入 `WD-EXP-SLOT-*`
12. 展示摘要

## 完成标准
- 每位参与专家都来自步骤 5 已选成员名单
- 每个需要专家参与的槽位都已有对应的 `WD-EXP-SLOT-*` 判断片段
- 每个参与槽位都给出：`参与槽位`、`决策类型`、`核心理由`、`关键证据引用`、`未决点`
- 所有 `新建` 结论都已附带不可复用 / 不可改造的证据说明
- 若存在 repowiki，相关槽位已纳入其内容并说明与 CTX 的对应关系
- 后续专家开始前只读取当前槽位相关的前序摘要，槽位内串行累计规则成立
- 所有需要的 `WD-EXP-SLOT-*` 片段已写入

## 输出
- 更新状态文件 checkpoints.step-9
- 所有所需 `WD-EXP-SLOT-*` 片段写入 working draft

## 结构化 JSON payload

```json
[
  {
    "slot": "2.1 方案设计",
    "decision_type": "改造",
    "rationale": "复用现有骨架并补齐专家分析。",
    "evidence_refs": ["CTX-01"],
    "open_questions": ["无"]
  }
]
```

## 门控
所有相关槽位分析未完成、任一必需槽位缺少 `决策类型` / `核心理由` / `关键证据引用`，或任一 `新建` 结论缺少不可复用 / 不可改造证据时，不能进入协作收敛，不得进入第 10 步。

## 回退信号
某些槽位的参与专家、必答问题、建议写法、支撑理由、补证信息变化。
