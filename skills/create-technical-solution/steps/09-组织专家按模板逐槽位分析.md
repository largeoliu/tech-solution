# 步骤 9：组织专家按模板逐槽位分析

## 输入
- 共享上下文
- 模板任务单
- repowiki 相关内容（如存在）
- working draft
- 状态文件

## 操作
1. 串行邀请每位专家（非并行），每位专家完整完成后再邀请下一位；后一位专家开始前必须先读取前面已写入的 `WD-EXP-*` 区块
2. 若步骤 6 确认存在 repowiki，相关槽位分析必须纳入其中与该槽位直接相关的内容；repowiki 只能补充判断，不得替代 CTX 编号 grounding
3. 每位专家按槽位逐项独立分析，不得提前收敛为统一口径；每个参与槽位都必须写出：是否参与、目标能力、现有候选、决策类型（`复用` / `改造` / `新建`）、建议写法、支撑理由、不可复用证据、不可改造证据、已使用CTX编号、未使用原因、依赖或冲突、仍需补证的信息、结论是否超出上下文支持
4. 若 `决策类型` 为 `新建`，`不可复用证据` 与 `不可改造证据` 必须显式回答“为什么不能复用现有对象”和“为什么不能在现有对象上改造”，且理由必须绑定 CTX 编号，不得只写风格偏好
5. 每位专家完成后写入 WD-EXP-[expert-slug] 区块，`expert-slug` 必须对应步骤 5 已选成员
6. 展示摘要

## 完成标准
- 每位参与专家都来自步骤 5 已选成员名单
- 每位参与专家都有对应 WD-EXP 区块
- 每个参与槽位都给出：是否参与、目标能力、现有候选、决策类型、建议写法、支撑理由、不可复用证据、不可改造证据、已使用CTX编号、未使用原因、依赖或冲突、仍需补证的信息、结论是否超出上下文支持
- 所有 `新建` 结论都已附带 `不可复用证据` / `不可改造证据`
- 若存在 repowiki，相关槽位已纳入其内容并说明与 CTX 的对应关系
- 后续专家开始前已读取前序 `WD-EXP-*`，串行累计规则成立
- 所有 WD-EXP 区块已写入

## 输出
- 更新状态文件 checkpoints.step-9
- 所有 WD-EXP-* 写入 working draft

## 门控
所有相关专家分析未完成，任一已选专家缺少 WD-EXP 区块，任一参与槽位缺少 `支撑理由` / `现有候选` / `决策类型`，或任一 `新建` 结论缺少 `不可复用证据` / `不可改造证据` 时，不能进入协作收敛，不得进入第 10 步。

## 回退信号
某些槽位的参与专家、必答问题、建议写法、支撑理由、补证信息变化。
