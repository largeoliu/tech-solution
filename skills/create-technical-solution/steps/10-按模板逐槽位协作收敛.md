# 步骤 10：按模板逐槽位协作收敛

## 输入
- 所有专家产物
- 共享上下文
- working draft
- 状态文件

## 操作
1. 进入本步骤前先通过 `python /path/to/run-step.py --state <状态文件路径>` 查看门禁。若未通过，优先根据 `repair_plan[]` 修复，并遵循其中 `depends_on_steps` 的前置顺序，再参考 `summary.recommended_repair_sequence`、`summary.recommended_rollback_step`、`summary.missing_artifacts` 与 `issues[*].repair_guidance` 确认修复效果后重试；通过后再进入收敛
2. 按 full 级别组织收敛，逐槽位逐个分析收敛。脚本支持多次 `--complete` 调用；公共主路径优先按 `slot_progress.recommended_slot` 单槽位推进，并使用 `--complete --ticket <ticket> --slot SLOT-XX`
3. 先运行 `python /path/to/run-step.py --state <状态文件路径> --advance` 进入本步。脚本返回 `slot_progress.completed_slots` 和 `slot_progress.recommended_slot`，建议按推荐顺序逐槽位提交
4. 获取单槽位 scaffold：`python /path/to/run-step.py --state <状态文件路径> --emit-json-scaffold --slot SLOT-XX`
5. 仅基于当前槽位已写入的 `WD-EXP-SLOT-*` 与相关 CTX，收敛成最小闭合集：
   - 目标能力
   - 候选方案对比
   - 选定路径
   - 选定写法
   - 关键证据引用
   - 建议落位槽位
   - 模板承载缺口
   - 未决问题
6. `候选方案对比` 至少覆盖三类路径：
   - 直接复用现有资产
   - 在现有资产上改造
   - 新建资产
   若某类路径不适用，必须写明不适用原因与证据
7. `选定写法` 是最终文档该槽位的正文来源，支持多段落 Markdown；应包含该槽位需要承载的全部正式内容，而非一句话摘要

**对比表格模板**（每个槽位必须包含）：
```markdown
### 槽位：{槽位名称}

#### 候选方案对比
| 路径 | 可行性 | 关键证据 | 选择理由 |
|------|--------|----------|----------|
| 复用 | ✅/❌ | CTX-XX: ... | ... |
| 改造 | ✅/❌ | CTX-XX: ... | ... |
| 新建 | ✅/❌ | CTX-XX: ... | ... |

#### 选定路径
- **路径**：{复用/改造/新建}
- **理由**：{一句话说明}
- **关键证据引用**：CTX-XX, CTX-YY
```
8. 记录当前槽位已核销的共享上下文、上下文冲突处理、未采纳理由、仍缺共享上下文、仍待决策；以下情况必须将槽位标记为阻塞，不得进入最终成稿：
   - 关键 CTX 仍缺失
   - 冲突未解
   - `新建` 路径缺少证明前两类路径不可接受的证据
   - 现有模板槽位无法承载必要结论且会导致结构越界
9. 将 `WD-SYN-SLOT-*` 按槽位增量写入 working draft 目录；标准入口是先运行 `python /path/to/run-step.py --state <状态文件路径> --advance` 进入本步骤并领取本步 ticket，再提交 **结构化 JSON payload**；脚本会先把 canonical decision truth 写入 `slots/SLOT-XX/decision.json`，并把对应产物追加到状态文件 `produced_artifacts` 和 `artifact_progress`
10. 当前步骤的 ticket 只允许写入 `WD-SYN-SLOT-*`；若 ticket 下发后 state、working draft、final document 或 block 范围变化，必须重新运行 `--advance` 获取新的 ticket
11. 禁止手工编辑状态文件中的 `slots`、`slot_count`、`task_slot_count`、`wd_exp_count`、`syn_slot_count` 去"收缩槽位"；也禁止删除所谓"多余 SLOT 目录"来修复不一致。模板与槽位不一致时，必须回到步骤 3 重建模板快照，再重做步骤 8/9/10。
12. 仅当所有非阻塞槽位都已写入本步骤所需收敛产物（`artifact_progress.WD-SYN-SLOT-*.completed_slots` 覆盖全部必需槽位），且无未标记阻塞槽位时，脚本自动把 `can_enter_step_11` 置为 true 并推进到步骤 11
13. `checkpoints.step-10.summary` 由脚本自动生成短格式流程摘要：
    - 增量提交时：`进行中；新增收敛；累计完成={K}/{N}；gate: step-10 continue`
    - 全部完成时：`完成；写入协作收敛；slots={N}；gate: step-11 ready`
14. 向用户展示摘要，包含累计进度
15. 在进入步骤 11 前，必须确保每个目标槽位的 `decision.json` 已先落盘到 working draft；不得先写最终文档，再回填收敛区块
16. `decision.json` 是步骤 10 的 canonical truth。最终成稿只吸收 `decision.json.selected_writeup`，并由脚本自动渲染 `decision.json.evidence_refs` 为 `依据:` 行；其余栏目（`候选方案对比`、`模板承载缺口`、`未决问题`）用于回溯、审计和回退，不进入正式文档

## 完成标准
- 每个槽位都有收敛纪要
- 候选方案对比已覆盖，且已包含 `复用 / 改造 / 新建` 最低比较集
- 进入最终成稿的槽位都能稳定映射回模板已有槽位
- 上下文冲突已处理，或已显式标记为阻塞槽位
- 关键 CTX 缺失的槽位已从最终成稿候选中剔除并记录原因
- 所有 `新建` 结论都已在 `关键证据引用` 中覆盖不可复用、不可改造与新建必要性的证据说明
- 每个受影响能力都已给出 `建议落位槽位`；若现有模板无法承载，已显式记录 `模板承载缺口`
- 已写入 `WD-SYN-SLOT-*`
- 摘要已向用户展示

## 输出
- 更新状态文件 checkpoints.step-10
- `slots/SLOT-XX/decision.json` canonical truth 写入 working draft
- 更新 `artifact_progress.WD-SYN-SLOT-*.completed_slots`

## 结构化 JSON payload

### 单槽位提交示例

```json
[
  {
    "slot": "2.1 方案设计",
    "target_capability": "收敛 2.1 方案设计 的最终写法。",
    "comparisons": [
      {"path": "复用", "feasibility": "❌", "evidence": "CTX-01", "reason": "不足"},
      {"path": "改造", "feasibility": "✅", "evidence": "CTX-01", "reason": "推荐"},
      {"path": "新建", "feasibility": "❌", "evidence": "CTX-01", "reason": "成本高"}
    ],
    "selected_path": "改造",
    "selected_writeup": "整体采用改造路径，在现有 OrderService 上扩展。\n\n新增字段 `spotRuleConfig` 存储比例配置，向后兼容。\n\n服务层增加策略分发逻辑，根据配置选择抽取方式。",
    "evidence_refs": ["CTX-01"],
    "template_gap": "无",
    "open_question": "无"
  }
]
```

## 门控
以下情况不能进入最终成稿，不得进入第 11 步：
- 协作收敛未完成或所需收敛区块未落盘
- 仍存在未标记阻塞的缺 CTX / 冲突槽位
- 任一 `new-build` 结论缺少前两类路径不可接受的证据
- 任一槽位存在未解决的 `模板承载缺口`

## 回退信号
某些槽位的共同结论、分歧点、选定写法、未采纳理由变化。
