# 步骤 10：按模板逐槽位协作收敛

## 输入
- 所有专家产物
- 共享上下文
- working draft
- 状态文件

## 操作
1. 进入本步骤前先通过 `python /path/to/run-step.py --state <状态文件路径>` 查看门禁。若未通过，优先根据 `repair_plan[]` 修复，并遵循其中 `depends_on_steps` 的前置顺序，再参考 `summary.recommended_repair_sequence`、`summary.recommended_rollback_step`、`summary.missing_artifacts` 与 `issues[*].repair_guidance` 确认修复效果后重试；通过后再进入收敛
2. 按 full 级别组织收敛，逐槽位逐个分析收敛，但所有槽位的收敛产物必须在**单次** `--complete` 提交流程中以完整 JSON payload 一次性提交，不得拆分成多次调用：
   仅基于当前槽位已写入的 `WD-EXP-SLOT-*` 与相关 CTX，收敛成最小闭合集：
     - 目标能力
     - 候选方案对比
     - 选定路径
     - 选定写法
     - 关键证据引用
     - 建议落位槽位
     - 模板承载缺口
     - 未决问题
3. `候选方案对比` 至少覆盖三类路径：
   - 直接复用现有资产
   - 在现有资产上改造
   - 新建资产
   若某类路径不适用，必须写明不适用原因与证据
4. `选定写法` 是最终文档该槽位的正文来源，支持多段落 Markdown；应包含该槽位需要承载的全部正式内容，而非一句话摘要

**对比表格模板**（每个槽位必须包含）：
```markdown
### 槽位：{槽位名称}

#### 候选方案对比
| 路径 | 可行性 | 关键证据 | 选择理由 |
|------|--------|----------|----------|
| 复用 | ✅/❌ | CTX-XX: ... | ... |
| 改造 | ✅/❌ | CTX-XX: ... | ... |
| 新建 | ✅/❌ | CTX-XX: ... | ... |

#### 选定路径
- **路径**：{复用/改造/新建}
- **理由**：{一句话说明}
- **关键证据引用**：CTX-XX, CTX-YY
```
4. 记录当前槽位已核销的共享上下文、上下文冲突处理、未采纳理由、仍缺共享上下文、仍待决策；以下情况必须将槽位标记为阻塞，不得进入最终成稿：
   - 关键 CTX 仍缺失
   - 冲突未解
   - `新建` 路径缺少证明前两类路径不可接受的证据
   - 现有模板槽位无法承载必要结论且会导致结构越界
5. 将 `WD-SYN-SLOT-*` 按槽位增量写入 working draft 目录，每个槽位的收敛结果写入 `slots/SLOT-XX/synthesis.md`；标准入口是先运行 `python /path/to/run-step.py --state <状态文件路径> --advance` 进入本步骤并领取本步 ticket，再提交 **结构化 JSON payload**；脚本会按槽位渲染 `WD-SYN-SLOT-*`，不得以最终文档替代收敛区块；并把对应产物追加到状态文件 `produced_artifacts`
6. 当前步骤的 ticket 只允许写入 `WD-SYN-SLOT-*`；若 ticket 下发后 state、working draft、final document 或 block 范围变化，必须重新运行 `--advance` 获取新的 ticket
7. 禁止手工编辑状态文件中的 `slots`、`slot_count`、`task_slot_count`、`wd_exp_count`、`syn_slot_count` 去“收缩槽位”；也禁止删除所谓“多余 SLOT 目录”来修复不一致。模板与槽位不一致时，必须回到步骤 3 重建模板快照，再重做步骤 8/9/10。
8. 仅当所有非阻塞槽位都已写入本步骤所需收敛产物，且无未标记阻塞槽位时，才把状态文件 `can_enter_step_11` 置为 true
9. 更新 `checkpoints.step-10` 时，`summary` 只能写短格式流程摘要，例如：`完成；写入 WD-SYN-SLOT-*；slots=19；gate: step-11 ready`
10. 向用户展示摘要
 11. 在进入步骤 11 前，必须确保 `WD-SYN-SLOT-*` 已先落盘到 working draft；不得先写最终文档，再回填收敛区块
12. `WD-SYN-SLOT-*` 是收敛过程产物，不是最终技术方案正文的直接拷贝源。最终成稿只吸收 `选定写法`，其余栏目（`候选方案对比`、`关键证据引用`、`模板承载缺口`、`未决问题`）用于回溯、审计和回退，不进入正式文档

## 完成标准
- 每个槽位都有收敛纪要
- 候选方案对比已覆盖，且已包含 `复用 / 改造 / 新建` 最低比较集
- 进入最终成稿的槽位都能稳定映射回模板已有槽位
- 上下文冲突已处理，或已显式标记为阻塞槽位
- 关键 CTX 缺失的槽位已从最终成稿候选中剔除并记录原因
- 所有 `新建` 结论都已在 `关键证据引用` 中覆盖不可复用、不可改造与新建必要性的证据说明
- 每个受影响能力都已给出 `建议落位槽位`；若现有模板无法承载，已显式记录 `模板承载缺口`
- 已写入 `WD-SYN-SLOT-*`
- 摘要已向用户展示

## 输出
- 更新状态文件 checkpoints.step-10
- `WD-SYN-SLOT-*` 写入 working draft

## 结构化 JSON payload

```json
[
  {
    "slot": "2.1 方案设计",
    "target_capability": "收敛 2.1 方案设计 的最终写法。",
    "comparisons": [
      {"path": "复用", "feasibility": "❌", "evidence": "CTX-01", "reason": "不足"},
      {"path": "改造", "feasibility": "✅", "evidence": "CTX-01", "reason": "推荐"},
      {"path": "新建", "feasibility": "❌", "evidence": "CTX-01", "reason": "成本高"}
    ],
    "selected_path": "改造",
    "selected_writeup": "整体采用改造路径，在现有 OrderService 上扩展。

新增字段 `spotRuleConfig` 存储比例配置，向后兼容。

服务层增加策略分发逻辑，根据配置选择抽取方式。",
    "evidence_refs": ["CTX-01"],
    "template_gap": "无",
    "open_question": "无"
  }
]
```

## 门控
<HARD-GATE>
步骤 10 必须在单次 `--complete` 中以结构化 JSON payload 一次性提交全部槽位的收敛结果。不得将槽位拆分到多次 `--complete` 调用中写入；写入首个 `WD-SYN-*` 后流程即推进到步骤 11。
</HARD-GATE>

以下情况不能进入最终成稿，不得进入第 11 步：
- 协作收敛未完成或所需收敛区块未落盘
- 仍存在未标记阻塞的缺 CTX / 冲突槽位
- 任一 `new-build` 结论缺少前两类路径不可接受的证据
- 任一槽位存在未解决的 `模板承载缺口`

## 回退信号
某些槽位的共同结论、分歧点、选定写法、未采纳理由变化。
