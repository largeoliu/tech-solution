# 技术方案分析指引

这个文档只定义“看什么”。它负责按方案类型提供参与成员建议、关键问题、易漏风险和评审重点，不负责最终文档结构。无论使用默认模板还是用户自定义模板，这份分析指引都可直接复用。

## 使用方式

先判断当前主题最像哪一类方案，再决定是否同时命中多类。

默认规则：

- 系统架构师始终参与
- 主题不简单时，通常加入可维护性专家
- 如果 `.architecture/members.yml` 中存在贴合主题的自定义专家，可以在核心成员之外追加

当一个主题同时命中多类时：

- 参与成员取并集并去重
- 必答问题取并集
- 易漏风险取并集
- 评审重点按风险高低排序

## 1. 新功能方案

适用信号：

- 新增用户能力、业务流程或系统接口
- 首次引入新的模块、服务或数据对象

优先参与成员：

- 系统架构师
- 领域专家
- 可维护性专家

按需补充：

- 涉及外部暴露接口、敏感数据或权限模型时加入安全专家
- 涉及高吞吐、低延迟或大规模访问时加入性能专家

必须回答的问题：

- 这个功能解决的核心问题是什么，非目标是什么
- 它归属哪个领域边界，是否引入新的领域概念
- 对现有接口、数据模型和调用链有什么影响
- 是否需要兼容旧行为、双写、迁移或灰度发布

最容易漏掉的风险：

- 领域概念命名不清，导致后续职责混乱
- 为了赶进度把新逻辑塞进旧模块，破坏边界
- 只写主流程，不写失败路径、回滚和兼容性处理

评审重点：

- 领域边界是否清楚
- 组件职责是否单一
- 新旧行为是否能安全共存

## 2. 重构或替换方案

适用信号：

- 替换现有模块、框架、依赖或实现方式
- 目标是降低复杂度、技术债或维护成本

优先参与成员：

- 系统架构师
- 可维护性专家

按需补充：

- 牵涉核心业务语义时加入领域专家
- 牵涉权限、密钥、审计或数据保护时加入安全专家
- 牵涉性能瓶颈时加入性能专家

必须回答的问题：

- 当前方案的主要痛点是什么，是否有可量化证据
- 替换边界在哪里，哪些部分必须保持兼容
- 是否支持渐进迁移、回滚和阶段性验收
- 新方案相对旧方案的复杂度、维护成本和风险如何变化

最容易漏掉的风险：

- 只写理想终态，不写过渡阶段
- 低估兼容层、数据迁移和回滚成本
- 把“想顺手优化的一切”都塞进同一次重构

评审重点：

- 替换边界和依赖方向是否清楚
- 迁移路径是否可执行、可回滚
- 重构范围是否被严格控制

## 3. 跨系统或平台能力方案

适用信号：

- 需要多个系统、服务、团队或运行环境协作
- 涉及统一平台能力、共享基础设施或跨边界数据流

优先参与成员：

- 系统架构师
- 领域专家
- 可维护性专家
- 安全专家
- 性能专家

必须回答的问题：

- 哪些系统参与，边界在哪里，谁拥有哪部分职责
- 接口、事件、数据同步和失败补偿如何设计
- 哪些依赖是同步的，哪些可以异步化
- 发生局部失败时，整体如何降级、重试或恢复

最容易漏掉的风险：

- 边界定义含糊，导致责任不清
- 只画成功链路，不写失败链路和补偿机制
- 把跨团队协作成本当作技术细节忽略掉

评审重点：

- 边界与职责划分是否稳定
- 依赖关系是否过深、过多或方向错误
- 数据一致性与故障恢复策略是否可信

## 4. 安全、合规或数据保护方案

适用信号：

- 涉及认证、授权、审计、密钥、隐私、数据分类、合规要求
- 核心目标是降低安全风险或满足监管要求

优先参与成员：

- 系统架构师
- 安全专家

按需补充：

- 涉及业务语义和权限边界时加入领域专家
- 涉及运维复杂度或长期维护时加入可维护性专家

必须回答的问题：

- 要保护的资产是什么，威胁面在哪里
- 认证、授权、审计和数据保护分别怎么落地
- 合规要求会影响哪些接口、存储、日志和访问流程
- 出现安全事件时，如何检测、追踪、隔离和恢复

最容易漏掉的风险：

- 只写访问控制，不写审计和追踪
- 忽略数据最小化、脱敏、保留周期和删除策略
- 把合规要求当注释，不落实到接口和流程约束

评审重点：

- 安全控制是否是设计内建约束，而不是后补
- 敏感数据流转是否可解释、可审计
- 权限边界和失败处理是否明确

## 5. 性能、容量或成本优化方案

适用信号：

- 目标是降低延迟、提升吞吐、增强容量或控制成本
- 已有系统在性能或资源利用上出现明显瓶颈

优先参与成员：

- 系统架构师
- 性能专家
- 可维护性专家

按需补充：

- 涉及缓存一致性、权限隔离或敏感数据时加入安全专家
- 涉及业务优先级和服务分层时加入领域专家

必须回答的问题：

- 当前瓶颈是什么，有哪些测量数据支撑
- 优化目标和目标值是什么
- 优化手段会如何影响一致性、复杂度、可维护性和成本
- 如果性能目标达不到，降级路径和兜底策略是什么

最容易漏掉的风险：

- 没有基线数据就开始优化
- 为局部性能提升引入全局复杂度
- 只写理想峰值，不写长期容量和运维成本

评审重点：

- 是否有可验证的性能目标和测量方法
- 优化是否破坏现有边界、职责和可维护性
- 成本节约与系统复杂度之间的权衡是否说清楚
