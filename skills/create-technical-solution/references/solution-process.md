# 技术方案标准产出流程

这个文档只定义“怎么产出”。它负责成员独立输入格式、协作收敛格式、标准信息块和质量门槛，不负责方案类型差异判断，也不负责具体模板落位。

## 1. 输入与前置约束

技术方案的输入可以来自：

- 一句话主题
- 需求描述
- 已有文档路径
- 代码、配置或系统现状说明

在开始写正文前，先补齐以下事实：

- 当前问题是什么，为什么现在要处理
- 这次要达成什么，不打算解决什么
- 业务、技术、时间、资源和原则约束分别是什么
- 影响范围涉及哪些模块、接口、数据流或团队

如果主题不明确，先澄清，再生成短横线风格文件名。清洗规则如下：

1. 去掉首尾空白，以及 `..`、`/`、`\` 这类路径控制字符
2. 将空格、下划线和常见分隔符折叠为 `-`
3. 转为小写，只保留 `[a-z0-9-]`
4. 去掉首尾 `-`
5. 如果清洗结果为空，要求用户提供更明确的标题

## 2. 文档职责与共享展示规则

本文件是技术方案过程契约的 canonical 来源，负责：

- canonical schema
- required fields
- completeness checks
- render order
- rollback decision logic

`references/progress-transparency.md` 负责：

- stage timing
- display-boundary wording
- user-facing phase protocol
- re-entry presentation framing

适用于全部用户可见产物的共享规则如下：

- `专家产物`、`协作收敛纪要`、`变更影响说明` 都必须先补齐本文件要求的必填字段，再对用户展示。
- 所有用户可见产物都必须按本文件定义的 canonical 顺序渲染，不得改序、删字段或改写字段含义。
- 长内容都采用摘要优先的展示组织方式，先给稳定结论，再补充支撑细节。
- 对外展示时的边界表述由 `references/progress-transparency.md` 统一约束，本文件不重复定义展示边界语言。

## 3. 参与成员独立输入格式

每个参与成员都必须按同一组字段输出。字段缺失时，不允许跳过，必须显式写“无新增风险”或“无额外未决问题”。

这里定义 `专家产物` 的 canonical 字段级 schema。

推荐格式如下：

```markdown
### [参与成员名称] - [角色头衔]

**视角说明**：
[引用该参与成员在 .architecture/members.yml 中的 perspective，并结合当前主题说明这次重点关注什么]

#### 设计目标
- [目标 1]
- [目标 2]

#### 关键约束
- [约束 1]
- [约束 2]

#### 推荐做法
- [建议 1]
- [建议 2]

#### 主要风险
- [风险 1：影响 + 触发条件 + 缓解方向]
- [风险 2：影响 + 触发条件 + 缓解方向]

#### 关键权衡
- [权衡 1：得到什么，失去什么]
- [权衡 2：得到什么，失去什么]

#### 未决问题
- [问题 1：还缺什么信息才能决策]
- [问题 2：需要谁补充信息]
```

独立输入阶段只做本角色的判断，不提前替别人收敛，也不直接复制现有模板句子。

### 对话内专家产物展示格式

对用户展示时，使用 `专家产物` 这一名称。它不是另一套新结构，而是对本节 canonical 字段的稳定渲染。

```markdown
### [参与成员名称] - [角色头衔]

**视角说明**：
[引用该参与成员在 .architecture/members.yml 中的 perspective，并结合当前主题说明这次重点关注什么]

[按本节 canonical 字段顺序渲染正文]
```

展示要求：

- `专家产物` 只代表当前参与成员的独立判断，不提前替其他参与成员收敛。

## 4. 协作收敛格式

参与成员独立输入完成后，必须统一收敛为下面这组字段。所有技术方案都用同一套收敛契约，不能按主题临时发明结构。

这里定义 canonical 内容名 `协作收敛` 的字段级 schema。对用户展示时，其 user-facing rendering name 为 `协作收敛纪要`；两者字段必须一一对应，不得增删映射。

```markdown
## 协作收敛

### 共同结论
- [团队一致认可的结论 1]
- [团队一致认可的结论 2]

### 争议点
- **[争议主题]**： [争议焦点]

### 候选方案对比
| 方案 | 核心做法 | 主要优点 | 主要缺点 | 适用前提 |
|------|----------|----------|----------|----------|
| A | [做法] | [优点] | [缺点] | [前提] |
| B | [做法] | [优点] | [缺点] | [前提] |

### 选定方向
- [最终选择]

### 选择理由
- [为什么选这个方向]
- [为什么不选其他方向]

### 原则冲突与取舍
- [涉及哪些项目原则]
- [冲突如何取舍]

### 未决问题与后续决策点
- [未决问题]
- [下一步由谁补齐什么信息]
```

协作收敛要求：

- 必须把争议显式写出来，不能只给最终答案。
- 每个候选方案都要写未选原因，避免只有一个推荐项。
- 如果存在原则冲突，必须说明冲突发生在哪一节、为什么这样取舍。

### 对话内协作纪要展示格式

对用户展示时，使用 `协作收敛纪要` 这一名称。它不是精简版，也不是会后摘要，必须完整映射本节 canonical 内容 `协作收敛`。

```markdown
## 协作收敛纪要

[按本节 canonical 字段顺序渲染正文]
```

展示要求：

- `协作收敛纪要` 必须与 canonical 内容 `协作收敛` 一一对应，不得只保留结论、不保留过程性结果。
- 被否决方案和未选原因必须保留在 `协作收敛纪要` 中，不能因已经定案而删除。
- 如存在强分歧或未决问题，必须继续保留，不得伪装为已达成一致。

## 5. 阶段回退与失效规则

当用户在阶段性展示后新增约束、修正目标、调整范围或补充关键事实时，先判断已有产物是否失效，再决定从哪个阶段重进。

### 触发条件

- 用户新增业务、技术、时间、资源或原则约束。
- 用户修正问题定义、目标、非目标或影响范围。
- 用户否定既有结论，或提供新信息使已有比较前提不再成立。

### 失效判断

- 先判断变化影响的是哪一类稳定结论：单个或多个 `专家产物`、`协作收敛纪要`、或仅最终模板映射内容。
- 受影响的既有内容必须显式标记为失效，不得继续当作当前有效结论复用。
- 未受影响且仍成立的稳定结论可以保留，但必须与失效内容清楚分隔。

### 最近受影响的阶段边界

- 如果变化只影响最终模板落位，不改变专家判断和收敛结论，则最近受影响的阶段边界为最终成稿阶段。
- 如果变化影响候选方案比较、选定方向、原则取舍或共同结论，则最近受影响的阶段边界为协作收敛阶段。
- 如果变化影响专家的设计目标、关键约束、推荐做法、主要风险、关键权衡或未决问题，则最近受影响的阶段边界为专家独立输入阶段。
- 阶段重进必须从 `最近受影响的阶段边界` 开始，不无故回退到更早阶段。

### 变更影响说明

发生回退、重算或阶段重进时，必须先整理出完整的 `变更影响说明`，再按 `references/progress-transparency.md` 的重进呈现约定对外展示。

```markdown
## 变更影响说明

- 触发变更：[用户新增或修正的信息]
- 受影响内容：[哪些专家产物、哪些协作结论、哪些模板映射内容失效]
- 最近受影响的阶段边界：[专家独立输入阶段 / 协作收敛阶段 / 最终成稿阶段]
- 保持有效的内容：[仍然成立、可直接复用的稳定结论]
- 下一步动作：[从哪个阶段重进，接下来会先展示什么]
```

要求：

- `变更影响说明` 必须明确写出失效范围，不得用模糊表述替代。
- `最近受影响的阶段边界` 必须显式写出，作为阶段重进的唯一起点说明。
- 如果无可复用内容，也要明确写“保持有效的内容：无”。
- `变更影响说明` 用于承接阶段回退与重进前的影响说明，不替代 `references/progress-transparency.md` 中的阶段化对外播报约定。

## 6. 标准信息块

协作收敛完成后，必须整理出一组模板无关的标准信息块。它们是内容契约，不等同于当前模板中的章节名。一个信息块可以独占一个章节，也可以和其他信息块共享同一个现有章节。

### 问题与背景

必须包含：

- 当前问题是什么
- 触发背景和当前现状
- 为什么现在要处理

### 目标与非目标

必须包含：

- 这次要达成的结果
- 明确不纳入本次方案的范围
- 成功判断标准或预期结果

### 约束与依赖

必须包含：

- 业务约束
- 技术约束
- 时间和资源约束
- 原则约束
- 关键依赖与影响范围

### 推荐方案

必须包含：

- 选定方向
- 核心做法
- 选择理由

### 备选方案与权衡

必须包含：

- 至少一个被认真比较过的备选方向
- 各候选方案的优点、缺点和适用前提
- 未选原因
- 原则冲突与取舍

### 详细设计

必须包含：

- 关键组件或模块
- 边界与职责
- 依赖关系
- 关键数据流
- 接口变更
- 兼容性影响
- 部署、发布、监控、可观测性、安全与合规要求

### 风险与缓解

必须包含：

- 主要风险
- 风险影响和发生概率判断
- 对应缓解措施

### 实施建议

必须包含：

- 建议里程碑
- 前置条件
- 分阶段实施顺序

### 评审关注点

必须包含：

- 最需要重点验证的设计点
- 当前争议点
- 高风险假设

### 未决问题

必须包含：

- 仍未做出的决定
- 还缺什么信息才能决策
- 由谁补齐这些信息

这些信息块如何落到当前模板，见 [template-adaptation.md](template-adaptation.md)。

## 7. 质量门槛

生成最终文档前，逐项自检：

- 结论是否具体到当前系统，而不是空泛表述
- 每份 `专家产物`、`协作收敛纪要` 和 `变更影响说明` 是否都在本文件要求的必填字段全部补齐后才展示
- 对外展示时，是否遵守 `references/progress-transparency.md` 定义的展示边界表述，避免展示 scratchpad、原始推理链路、未成形草稿或原始争论记录
- 长内容是否按摘要优先方式展示，而不是直接倾倒原始长文
- 目标和非目标是否清楚分开
- 每个标准信息块是否都已生成，而不是只保留零散结论
- 每个备选方案是否都写了未选原因
- `协作收敛纪要` 是否保留被否决方案及其未选原因，而不是只留最终选定方向
- 每个风险是否都带影响、概率判断和缓解方向
- 是否已经写出边界与职责、依赖关系，而不是只写模块名称
- 是否把原则约束写进约束相关内容，而不是只放在脑中
- 未决问题是否写清楚缺什么信息、由谁补齐、何时决策
- 用户中途打断或补充约束时，是否清楚写出失效范围、`最近受影响的阶段边界` 和重新开始的阶段
- 回退或重进前，是否先输出 `变更影响说明`
- 最终内容是否已落到当前模板结构中，而不是退回默认模板章节名
- 是否避免新增用户模板没有定义的一级章节
- 是否避免把原模板重写回默认模板结构

如果上述任一项不满足，先修正，再生成或更新技术方案文件。
