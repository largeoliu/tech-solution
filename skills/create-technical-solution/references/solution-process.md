# 技术方案标准产出流程

这个文档只定义“怎么产出”。它负责模板任务单、专家按模板逐槽位分析、按模板逐槽位协作收敛、最终最低语义义务和质量门槛，不负责方案类型差异判断，也不负责具体模板落位。主 skill 提供摘要式闭环；本文件提供完整版细则。

## 1. 输入与前置约束

技术方案的输入可以来自：

- 一句话主题
- 需求描述
- 已有文档路径
- 代码、配置或系统现状说明

在开始写正文前，先补齐以下事实：

- 当前问题是什么，为什么现在要处理
- 这次要达成什么，不打算解决什么
- 业务、技术、时间、资源和原则约束分别是什么
- 影响范围涉及哪些模块、接口、数据流或团队

如果主题不明确，先澄清，再生成短横线风格文件名。清洗规则如下：

1. 去掉首尾空白，以及 `..`、`/`、`\` 这类路径控制字符
2. 将空格、下划线和常见分隔符折叠为 `-`
3. 转为小写，只保留 `[a-z0-9-]`
4. 去掉首尾 `-`
5. 如果清洗结果为空，要求用户提供更明确的标题

## 2. 文档职责与共享展示规则

本文件补充 `skills/create-technical-solution/SKILL.md`，只展开产出细则，不定义顶层执行顺序，也不替代主 skill 的权威入口角色。本文负责：

- `模板任务单` 的 canonical schema
- `专家按模板逐槽位分析` 的 canonical schema
- `按模板逐槽位协作收敛` 的 canonical schema
- 最终文档的最低语义义务定义
- 完整质量门槛
- 阶段回退或重新进入时必须保留的产物字段要求

`references/progress-transparency.md` 负责：

- stage timing
- display-boundary wording
- user-facing phase protocol
- re-entry presentation framing

适用于全部用户可见产物的共享规则如下：

- `共享上下文清单`、`模板任务单`、`专家按模板逐槽位分析`、`按模板逐槽位协作收敛`、`变更影响说明` 都必须先补齐本文件要求的必填字段，再写入 working draft，再对用户展示。
- 对外展示时，默认只展示稳定摘要，并使用 `详见 working draft：WD-CTX（共享上下文清单）` 这类稳定引用指向完整内容。
- 阶段回退或重算时，先更新 working draft 中对应区块，再对外展示 `变更影响说明`。
- 最终文档生成后，先执行吸收检查，再删除 working draft。
- `模板任务单`、`专家按模板逐槽位分析`、`按模板逐槽位协作收敛` 这三类槽位产物都必须保持当前模板顺序、当前槽位语义和本文件定义的字段顺序，不得删字段、改写字段含义或跳过必填项；`变更影响说明` 不要求按模板槽位顺序展开，但不得与这些槽位产物相矛盾。
- `模板任务单`、`专家按模板逐槽位分析`、`按模板逐槽位协作收敛` 这三类槽位产物都不得把模板内容改写成模板外的通用标题骨架；只能围绕当前模板已有槽位展开，不得重写成额外的 generic headings。
- 对外展示时的边界表述由 `references/progress-transparency.md` 统一约束，本文件不重复定义展示边界语言。

### 共享上下文清单

进入 `模板任务单` 前，必须先形成一份 `共享上下文清单`。这是后续槽位绑定、专家分析、协作收敛和最终成稿的唯一共享证据池；后续阶段只能消费这份清单中已经显式列出的内容，不能把未记录的背景常识当作隐含前提。

这里定义 `共享上下文清单` 的最小 schema。

```markdown
## 共享上下文清单

- `上下文编号`： [稳定编号，供后续槽位和专家产物引用]
- `来源`： [来自原则文档、代码、配置、repowiki、既有实现、相关文档或外部约束]
- `结论或约束`： [该来源支持的稳定结论、已知事实或必须遵守的约束]
- `适用槽位`： [当前模板中会消费这条上下文的槽位标识；无则显式写无]
- `可信度或缺口`： [已验证 / 待补证 / 存在冲突 / 缺口说明]
```

要求：

- `共享上下文清单` 必须先于 `模板任务单` 完成，不能在专家分析过程中临时补造。
- 每条上下文都必须有明确来源，不允许写“常识”“经验判断”这类不可追溯来源。
- `适用槽位` 必须与当前模板槽位相匹配；如果无法安全映射到任何槽位，应显式标记并说明原因。
- 如果关键槽位没有足够共享上下文支撑，必须先标记缺口，再停止进入 `模板任务单` 或后续阶段。

## 3. 模板任务单

进入专家分析前，必须先把当前生效模板拆成稳定槽位，并生成 `模板任务单`。这是全部后续分析与收敛的唯一可见骨架；用户请求的 `专家罗盘` 必须嵌入到 `每位专家必答问题` 中，而不是再额外生成一层模板外结构。

这里定义 `模板任务单` 的 canonical schema。

```markdown
## 模板任务单

### [按当前模板顺序的槽位标题]
- `槽位标识`： [稳定标识，便于在后续阶段引用]
- `槽位语义`： [这个槽位在当前模板中负责承载什么信息]
- `可填写边界`： [允许写什么，不允许写什么]
- `本槽位必须消费的共享上下文`： [必须引用的共享上下文编号；无则显式写无]
- `参与专家`： [需要参与该槽位的成员；无则显式写无]
- `每位专家必答问题`：
  - [专家 A：围绕该槽位必须回答的问题]
  - [专家 B：围绕该槽位必须回答的问题]
- `禁止越界项`： [容易被写到别的槽位或模板外的内容]
- `依赖槽位`： [本槽位依赖哪些前置槽位；无则显式写无]
- `缺失即停止的上下文`： [缺少哪些共享上下文时不得继续推进该槽位]
- `缺口或阻塞项`： [缺什么信息会阻止安全落位；无则显式写无]
```

要求：

- `模板任务单` 必须严格按当前模板顺序列出槽位，不得自行换序。
- `每位专家必答问题` 是唯一承载专家罗盘的地方；不要再追加模板外的独立“专家罗盘”层。
- 如果某些信息不适用当前槽位，必须显式标记不适用或无，不要强行塞入别的槽位。
- `禁止越界项` 必须明确提醒哪些常见内容虽然重要，但属于其他已有槽位，或属于最终文档中依法承载该语义义务的现有位置。
- 任何槽位如果没有足够语义支撑，必须标记为阻塞并停止后续自动成稿。
- `本槽位必须消费的共享上下文` 只能引用 `共享上下文清单` 中已有编号，不允许自由发明新证据项。
- `缺失即停止的上下文` 必须点名关键缺口，避免后续阶段用泛化经验把该槽位伪装成已完成。

## 4. 专家按模板逐槽位分析

每个参与专家都必须围绕 `模板任务单` 逐槽位输出，不允许再回到通用专家壳。专家不能再以 `设计目标`、`关键约束`、`主要风险`、`关键权衡` 等通用标题组织自己的主结构。只能围绕模板已有槽位回答。如果某个槽位与该专家无关，应明确标记无贡献，而不是自由发挥到别的结构里。

这里定义 `专家按模板逐槽位分析` 的 canonical schema。

```markdown
## 专家按模板逐槽位分析

### [专家名称] - [角色头衔]

#### [槽位标识] [槽位标题]
- `是否参与该槽位`： [参与 / 不参与，并说明原因]
- `建议写法或建议内容`： [建议直接落入该槽位的内容；无则显式写无]
- `支撑理由`： [基于该专家视角的依据、证据或推理]
- `已使用的共享上下文编号`： [引用本次判断实际消费的共享上下文编号；无则显式写无]
- `未使用原因`： [对任务单要求但本轮未使用的共享上下文，说明为什么未使用；无则显式写无]
- `结论是否超出上下文支持`： [没有超出 / 部分超出 / 明显超出，并说明原因]
- `与其他槽位的依赖或冲突`： [引用相关槽位标识；无则显式写无]
- `仍需补证的信息`： [还缺什么信息才能稳定写入；无则显式写无]
```

要求：

- `专家按模板逐槽位分析` 只做本角色判断，不提前替别人收敛，也不直接重写最终文档。
- 每位专家都必须按同一槽位顺序回答，保持与 `模板任务单` 一致。
- `建议写法或建议内容` 必须可回填到现有模板槽位，不能输出模板外可见结构。
- 如果某个槽位不适用，该专家仍要保留该槽位并显式写 `是否参与该槽位：不参与`。
- `已使用的共享上下文编号` 必须与 `模板任务单` 中该槽位要求消费的上下文编号相对应。
- 如果专家建议无法引用共享上下文编号，或 `结论是否超出上下文支持` 不是“没有超出”，必须把该槽位继续标记为待补证，而不是直接进入收敛定稿。

## 5. 按模板逐槽位协作收敛

全部专家完成逐槽位分析后，必须按同一槽位顺序进行收敛。收敛单位是槽位，不是通用主题块。被否决选项、分歧和未采纳理由必须继续可见；不适用的元数据必须显式标记，而不是被强行改写成有内容的结论。

这里定义 `按模板逐槽位协作收敛` 的 canonical schema。对用户展示时，其 user-facing rendering name 为 `协作收敛纪要`；两者字段必须一一对应，不得增删映射。

```markdown
## 按模板逐槽位协作收敛

### [槽位标识] [槽位标题]
- `共同结论`： [当前槽位达成一致的内容；无则显式写无]
- `分歧点`： [仍存在的意见差异；无则显式写无]
- `选定写法`： [准备写入最终模板的稳定文本或结构]
- `本槽位已核销的共享上下文`： [已被验证为支撑该槽位最终写法的共享上下文编号]
- `上下文冲突如何处理`： [若共享上下文之间或专家解释之间有冲突，说明取舍方式；无则显式写无]
- `未采纳理由`： [被否决写法为什么不采用；无则显式写无]
- `仍缺哪条共享上下文`： [仍缺失但会影响最终落位的共享上下文编号或说明；无则显式写无]
- `仍待决策`： [当前槽位还不能定的点；无则显式写无]
```

要求：

- `按模板逐槽位协作收敛` 必须与 `模板任务单` 保持同样的槽位顺序。
- 被否决方案和 `未采纳理由` 必须保留，不能因已有主选项而删除。
- 如存在强分歧或未决问题，必须继续保留，不得伪装为已达成一致。
- `选定写法` 只能落回当前模板已有槽位，不得外溢为模板外可见结构。
- `本槽位已核销的共享上下文` 必须能覆盖最终 `选定写法` 所依赖的关键事实和约束。
- 如果 `仍缺哪条共享上下文` 不为无，或上下文冲突未处理完毕，该槽位不得进入最终成稿。

## 6. 阶段回退与失效规则

当用户在阶段性展示后新增约束、修正目标、调整范围或补充关键事实时，先判断已有产物是否失效，再决定从哪个阶段重进。

### 触发条件

- 用户新增业务、技术、时间、资源或原则约束。
- 用户修正问题定义、目标、非目标或影响范围。
- 用户否定既有结论，或提供新信息使已有比较前提不再成立。

### 失效判断

- 先判断变化影响的是哪一类稳定结论：单个或多个 `模板任务单` 槽位、`专家按模板逐槽位分析`、`协作收敛纪要`、或仅最终模板映射内容。
- 受影响的既有内容必须显式标记为失效，不得继续当作当前有效结论复用。
- 未受影响且仍成立的稳定结论可以保留，但必须与失效内容清楚分隔。

### 最近受影响的阶段边界

- 如果变化只影响最终模板落位，不改变 `模板任务单`、专家判断和槽位收敛结论，则最近受影响的阶段边界为 `严格模板成稿阶段`。
- 如果变化影响某些槽位的共同结论、分歧点、选定写法或未采纳理由，则最近受影响的阶段边界为 `按模板逐槽位协作收敛阶段`。
- 如果变化影响某些槽位的参与专家、必答问题、建议写法、支撑理由或补证信息，则最近受影响的阶段边界为 `专家按模板逐槽位分析阶段`。
- 如果变化导致模板槽位定义、槽位边界、参与专家分配或阻塞条件本身失效，则最近受影响的阶段边界为 `模板任务单阶段`。
- 阶段重进必须从 `最近受影响的阶段边界` 开始，不无故回退到更早阶段。

### 变更影响说明

发生回退、重算或阶段重进时，必须先整理出完整的 `变更影响说明`，再按 `references/progress-transparency.md` 的重进呈现约定对外展示。

```markdown
## 变更影响说明

- 触发变更：[用户新增或修正的信息]
- 受影响内容：[哪些模板任务单槽位、哪些专家槽位分析、哪些协作收敛结论、哪些模板映射内容失效]
- 最近受影响的阶段边界：[模板任务单阶段 / 专家按模板逐槽位分析阶段 / 按模板逐槽位协作收敛阶段 / 严格模板成稿阶段]
- 保持有效的内容：[仍然成立、可直接复用的稳定结论]
- 下一步动作：[从哪个阶段重进，接下来会先展示什么]
```

要求：

- `变更影响说明` 必须明确写出失效范围，不得用模糊表述替代。
- `最近受影响的阶段边界` 必须显式写出，作为阶段重进的唯一起点说明。
- 如果无可复用内容，也要明确写“保持有效的内容：无”。
- `变更影响说明` 用于承接阶段回退与重进前的影响说明，不替代 `references/progress-transparency.md` 中的阶段化对外播报约定。

## 7. 最终文档的最低语义义务

协作收敛完成后，最终文档仍必须覆盖下面这组最低语义义务。它们只是最终成稿前的检查清单。不是模板外的中间层，也不是要求另起一套通用章节名。它们只是检查当前模板最终成稿时，是否已经完整承载了这些最少语义。

### 问题与背景

必须包含：

- 当前问题是什么
- 触发背景和当前现状
- 为什么现在要处理

### 目标与非目标

必须包含：

- 这次要达成的结果
- 明确不纳入本次方案的范围
- 成功判断标准或预期结果

### 约束与依赖

必须包含：

- 业务约束
- 技术约束
- 时间和资源约束
- 原则约束
- 关键依赖与影响范围

### 推荐方案

必须包含：

- 选定方向
- 核心做法
- 选择理由

### 备选方案与权衡

必须包含：

- 至少一个被认真比较过的备选方向
- 各候选方案的优点、缺点和适用前提
- 未选原因
- 原则冲突与取舍

### 详细设计

必须包含：

- 关键组件或模块
- 边界与职责
- 依赖关系
- 关键数据流
- 接口变更
- 兼容性影响
- 部署、发布、监控、可观测性、安全与合规要求

### 风险与缓解

必须包含：

- 主要风险
- 风险影响和发生概率判断
- 对应缓解措施

### 实施建议

必须包含：

- 建议里程碑
- 前置条件
- 分阶段实施顺序

### 评审关注点

必须包含：

- 最需要重点验证的设计点
- 当前争议点
- 高风险假设

### 未决问题

必须包含：

- 仍未做出的决定
- 还缺什么信息才能决策
- 由谁补齐这些信息

这些最低语义义务如何落到当前模板，见 [template-adaptation.md](template-adaptation.md)。

## 8. 质量门槛

生成最终文档前，先满足主 skill 可见的最小质量门槛，再逐项自检本节的完整质量门槛：

- 结论是否具体到当前系统，而不是空泛表述
- 每份 `模板任务单`、`专家按模板逐槽位分析`、`协作收敛纪要` 和 `变更影响说明` 是否都在本文件要求的必填字段全部补齐后才展示
- 对外展示时，是否遵守 `references/progress-transparency.md` 定义的展示边界表述，避免展示 scratchpad、原始推理链路、未成形草稿或原始争论记录
- 长内容是否按摘要优先方式展示，而不是直接倾倒原始长文
- `共享上下文清单` 是否先于 `模板任务单` 形成，并为每条上下文提供稳定来源与适用槽位
- 目标和非目标是否清楚分开
- 当前模板的每个槽位是否都已有 `模板任务单`、专家分析和收敛记录，而不是只保留零散结论
- 每个槽位的 `本槽位必须消费的共享上下文`、`已使用的共享上下文编号` 和 `本槽位已核销的共享上下文` 是否形成完整可追溯链路
