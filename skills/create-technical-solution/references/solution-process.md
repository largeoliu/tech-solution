# 技术方案标准产出流程

这个文档只定义“怎么产出”。它负责成员独立输入格式、协作收敛格式、标准信息块和质量门槛，不负责方案类型差异判断，也不负责具体模板落位。

## 1. 输入与前置约束

技术方案的输入可以来自：

- 一句话主题
- 需求描述
- 已有文档路径
- 代码、配置或系统现状说明

在开始写正文前，先补齐以下事实：

- 当前问题是什么，为什么现在要处理
- 这次要达成什么，不打算解决什么
- 业务、技术、时间、资源和原则约束分别是什么
- 影响范围涉及哪些模块、接口、数据流或团队

如果主题不明确，先澄清，再生成短横线风格文件名。清洗规则如下：

1. 去掉首尾空白，以及 `..`、`/`、`\` 这类路径控制字符
2. 将空格、下划线和常见分隔符折叠为 `-`
3. 转为小写，只保留 `[a-z0-9-]`
4. 去掉首尾 `-`
5. 如果清洗结果为空，要求用户提供更明确的标题

## 2. 成员独立输入格式

每个参与成员都必须按同一组字段输出。字段缺失时，不允许跳过，必须显式写“无新增风险”或“无额外未决问题”。

推荐格式如下：

```markdown
### [成员名称] - [角色头衔]

**视角说明**：
[引用该成员在 .architecture/members.yml 中的 perspective，并结合当前主题说明这次重点关注什么]

#### 设计目标
- [目标 1]
- [目标 2]

#### 关键约束
- [约束 1]
- [约束 2]

#### 推荐做法
- [建议 1]
- [建议 2]

#### 主要风险
- [风险 1：影响 + 触发条件 + 缓解方向]
- [风险 2：影响 + 触发条件 + 缓解方向]

#### 关键权衡
- [权衡 1：得到什么，失去什么]
- [权衡 2：得到什么，失去什么]

#### 未决问题
- [问题 1：还缺什么信息才能决策]
- [问题 2：需要谁补充信息]
```

独立输入阶段只做本角色的判断，不提前替别人收敛，也不直接复制现有模板句子。

## 3. 协作收敛格式

成员独立输入完成后，必须统一收敛为下面这组字段。所有技术方案都用同一套收敛契约，不能按主题临时发明结构。

```markdown
## 协作收敛

### 共同结论
- [团队一致认可的结论 1]
- [团队一致认可的结论 2]

### 争议点
- **[争议主题]**： [争议焦点]

### 候选方案对比
| 方案 | 核心做法 | 主要优点 | 主要缺点 | 适用前提 |
|------|----------|----------|----------|----------|
| A | [做法] | [优点] | [缺点] | [前提] |
| B | [做法] | [优点] | [缺点] | [前提] |

### 选定方向
- [最终选择]

### 选择理由
- [为什么选这个方向]
- [为什么不选其他方向]

### 原则冲突与取舍
- [涉及哪些项目原则]
- [冲突如何取舍]

### 未决问题与后续决策点
- [未决问题]
- [下一步由谁补齐什么信息]
```

协作收敛要求：

- 必须把争议显式写出来，不能只给最终答案。
- 每个候选方案都要写未选原因，避免只有一个推荐项。
- 如果存在原则冲突，必须说明冲突发生在哪一节、为什么这样取舍。

## 4. 标准信息块

协作收敛完成后，必须整理出一组模板无关的标准信息块。它们是内容契约，不等同于当前模板中的章节名。一个信息块可以独占一个章节，也可以和其他信息块共享同一个现有章节。

### 问题与背景

必须包含：

- 当前问题是什么
- 触发背景和当前现状
- 为什么现在要处理

### 目标与非目标

必须包含：

- 这次要达成的结果
- 明确不纳入本次方案的范围
- 成功判断标准或预期结果

### 约束与依赖

必须包含：

- 业务约束
- 技术约束
- 时间和资源约束
- 原则约束
- 关键依赖与影响范围

### 推荐方案

必须包含：

- 选定方向
- 核心做法
- 选择理由

### 备选方案与权衡

必须包含：

- 至少一个被认真比较过的备选方向
- 各候选方案的优点、缺点和适用前提
- 未选原因
- 原则冲突与取舍

### 详细设计

必须包含：

- 关键组件或模块
- 边界与职责
- 依赖关系
- 关键数据流
- 接口变更
- 兼容性影响
- 部署、发布、监控、可观测性、安全与合规要求

### 风险与缓解

必须包含：

- 主要风险
- 风险影响和发生概率判断
- 对应缓解措施

### 实施建议

必须包含：

- 建议里程碑
- 前置条件
- 分阶段实施顺序

### 评审关注点

必须包含：

- 最需要重点验证的设计点
- 当前争议点
- 高风险假设

### 未决问题

必须包含：

- 仍未做出的决定
- 还缺什么信息才能决策
- 由谁补齐这些信息

这些信息块如何落到当前模板，见 [template-adaptation.md](template-adaptation.md)。

## 5. 质量门槛

生成最终文档前，逐项自检：

- 结论是否具体到当前系统，而不是空泛表述
- 目标和非目标是否清楚分开
- 每个标准信息块是否都已生成，而不是只保留零散结论
- 每个备选方案是否都写了未选原因
- 每个风险是否都带影响、概率判断和缓解方向
- 是否已经写出边界与职责、依赖关系，而不是只写模块名称
- 是否把原则约束写进约束相关内容，而不是只放在脑中
- 未决问题是否写清楚缺什么信息、由谁补齐、何时决策
- 最终内容是否已落到当前模板结构中，而不是退回默认模板章节名
- 是否避免新增用户模板没有定义的一级章节
- 是否避免把原模板重写回默认模板结构

如果上述任一项不满足，先修正，再生成或更新技术方案文件。
