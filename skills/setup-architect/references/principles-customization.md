# 定制架构原则

根据你的项目框架和技术栈定制这个原则文档： `.architecture/principles.md`

## 编辑原则

初始化 `.architecture/principles.md` 时，先复制 [../templates/principles-template.md](../templates/principles-template.md) 作为起点。完成初始化后，再按项目现实补充、收紧或细化原则。

**添加原则**：
1. 从 `principles-template.md` 末尾的 HTML 注释指引复制一个项目特定原则小节
2. 结合你的项目现实填写标题、描述、理由和应用
3. 添加到 `.architecture/principles.md` 的合适位置
4. 确保它能直接指导技术方案、实施计划和架构评审

**移除原则**：
1. 只移除明显不适用的项目特定补充原则
2. 不要删除下方“核心原则（保留这些）”中的基础原则

**修改原则**：
1. 编辑 `.architecture/principles.md` 中现有条目，使其与项目现实一致
2. 至少覆盖模块边界与依赖方向、API / 事件 / 数据边界、测试和验证基线、安全与合规底线，以及技术方案和实施计划的决策标准


## 核心原则（保留这些）

这些核心原则应该为所有项目保留：
- **宜居代码**：围绕长期维护和开发体验设计代码库
- **清晰优于炫技**：优先选择简单清晰的设计
- **关注点分离**：明确组件职责边界和依赖关系
- **可演化性**：支持系统随时间安全演进
- **可观测性**：为运行行为、性能和状态提供可见性
- **设计即安全**：将安全作为架构内建约束
- **领域中心设计**：让架构反映并服务于问题领域
- **务实的简洁性**：优先选择实用、可工作的方案
- **变更影响意识**：在决策前显式评估影响范围、可逆性和时机

这些核心原则已内置在 [../templates/principles-template.md](../templates/principles-template.md) 中，可直接用来初始化 `.architecture/principles.md`。

**项目定制应补充或收紧这些原则，不要删除核心原则。**
