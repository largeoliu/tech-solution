# 架构原则

本文档用于为后续技术方案编写与评审提供项目上下文和判断基线。
它不应写成 README、系统盘点或运维手册，而应聚焦那些会持续影响方案取舍的项目事实、风险和约束。
核心原则必须保留，但要翻译成当前项目里的具体判断规则。

## 文档目的与使用方式

**相关原则**：宜居代码、清晰优于炫技、变更影响意识

### 本项目事实
- 在这里概括本文档服务的读者、使用时机，以及哪些场景必须引用本文档。
- 说明它与 `.architecture/technical-solutions/` 中方案文档的关系，但不要重复方案模板本身。

### 为什么重要
- 解释如果没有统一判断基线，团队会在哪些评审、协作或落地环节反复争论或做出不一致决策。

### 对方案与评审的要求
- 方案必须引用这里定义的关键约束，并说明哪些原则直接影响本次设计。
- 评审时优先检查方案是否基于项目现实做判断，而不是套用通用最佳实践口号。

## 业务域与项目语义

**相关原则**：领域中心设计、宜居代码

### 本项目事实
- 在这里写清核心业务对象、关键业务流程、统一术语，以及容易混淆的项目内词汇。
- 明确哪些概念是核心域，哪些只是支撑域或外围集成概念。

### 为什么重要
- 说明术语不统一、模型边界模糊时，会如何导致错误抽象、职责漂移或方案误判。

### 对方案与评审的要求
- 方案必须使用项目内统一语言描述模块、接口和数据对象。
- 若引入新概念，必须解释它与现有业务语义的关系，以及为何不能复用既有模型。

## 模块边界与依赖方向

**相关原则**：关注点分离、可演化性、清晰优于炫技

### 本项目事实
- 在这里说明当前系统如何划分模块、服务、包或目录层次，以及哪些依赖方向是硬约束。
- 标出历史包袱、共享层、跨模块调用习惯和禁止继续扩散的耦合点。

### 为什么重要
- 解释一旦边界失守，会在哪些地方放大变更成本、降低可测试性或阻碍后续演进。

### 对方案与评审的要求
- 方案必须明确新增能力落在哪个模块边界内，以及为何不应放到其他位置。
- 涉及跨边界调用时，必须说明依赖方向是否被打破、是否需要适配层，以及如何控制扩散。

## 数据、接口与事件边界

**相关原则**：关注点分离、可观测性、设计即安全

### 本项目事实
- 在这里说明核心数据归属、外部接口、内部接口、事件流转和兼容性约束。
- 标出哪些字段、事件、契约或数据副本是事实来源，哪些只是投影或缓存。

### 为什么重要
- 说明边界不清会如何引发重复写入、契约漂移、数据口径冲突或排障困难。

### 对方案与评审的要求
- 方案必须说明数据写入点、读取路径、接口兼容策略和事件语义。
- 涉及新接口或新事件时，必须写清版本、幂等、失败处理和观测方式。

## 质量属性与治理底线

**相关原则**：可观测性、设计即安全、务实的简洁性

### 本项目事实
- 在这里写清性能、稳定性、安全、测试、发布、合规和审计方面已经存在的底线要求。
- 记录当前团队真正执行的验证门槛，而不是理想化口号。

### 为什么重要
- 解释哪些质量属性是业务承诺、外部约束或历史事故后的硬要求，为什么不能在方案里被弱化。

### 对方案与评审的要求
- 方案必须说明如何验证关键质量属性，缺失验证路径时不能宣称可落地。
- 若需要偏离既有治理底线，必须明确风险、补偿措施和审批前提。

## 当前必须尊重的项目现实

**相关原则**：宜居代码、务实的简洁性、变更影响意识

### 本项目事实
- 在这里写清当前阶段不可忽视的现实约束，例如团队能力、交付窗口、遗留系统、运行环境或预算限制。
- 区分短期必须接受的现实与中长期计划改善的问题。

### 为什么重要
- 说明忽视这些现实会如何导致方案虽然正确却无法实施，或让改造成本与组织成本失控。

### 对方案与评审的要求
- 方案必须显式说明哪些设计是顺应当前现实，哪些是为后续演进预留接口。
- 评审时要检查计划是否匹配团队与系统当前承载能力，而不是只看目标状态是否漂亮。

## 方案编写与评审准绳

**相关原则**：清晰优于炫技、可演化性、变更影响意识

### 本项目事实
- 在这里总结本项目评估方案时最看重的决策维度、常见否决原因和必须回答的问题。
- 可以列出历史上反复出现的坏味道，例如抽象过度、跳过迁移路径、忽略回滚与观测。

### 为什么重要
- 说明统一评审准绳如何帮助团队更快识别高风险方案，并减少只停留在实现偏好的争论。

### 对方案与评审的要求
- 方案必须写清目标、边界、权衡、验证、迁移和回滚；缺一项就不算完整。
- 评审意见应优先围绕项目约束、变更影响和落地可行性，不要只给抽象偏好判断。

## 原则映射表

| 核心原则 | 主要落点章节 | 本项目中的约束含义 |
| --- | --- | --- |
| 宜居代码 | 文档目的与使用方式 | 规则要帮助长期协作与维护，不为展示理论完整性而增加阅读负担。 |
| 清晰优于炫技 | 模块边界与依赖方向 | 设计优先选择团队能稳定理解和执行的直接方案，避免炫技式抽象。 |
| 关注点分离 | 数据、接口与事件边界 | 模块、数据和接口职责要可解释、可验证，避免隐式共享与边界漂移。 |
| 可演化性 | 方案编写与评审准绳 | 方案需要说明演进路径、兼容策略和后续调整成本，不能只满足当下。 |
| 可观测性 | 质量属性与治理底线 | 新方案必须具备必要日志、指标、追踪或审计能力，便于定位问题。 |
| 设计即安全 | 质量属性与治理底线 | 安全与合规要求要体现在接口、数据和权限设计里，而不是上线前补洞。 |
| 领域中心设计 | 业务域与项目语义 | 模型、命名和边界要服务真实业务语义，避免技术结构压过领域表达。 |
| 务实的简洁性 | 当前必须尊重的项目现实 | 方案要尊重团队与系统现状，用最小可行复杂度解决真实问题。 |
| 变更影响意识 | 方案编写与评审准绳 | 每个重要决策都要说明影响范围、可逆性、时机与协作成本。 |
