# 架构原则

本文档概述了指导本项目所有设计决策的核心架构原则。在做架构决策、评估设计和进行评审时，应参考这些原则。

## 来自软件先驱的智慧

> “任何傻瓜都能写出计算机看得懂的代码。优秀的程序员写的是人类看得懂的代码。”<br>
> — **Martin Fowler**

> “糟糕的程序员担心代码。优秀的程序员担心数据结构及其相互关系。”<br>
> — **Linus Torvalds**

> “软件的核心，在于它能够为用户解决与领域相关的问题。”<br>
> — **Eric Evans**

> “我并不是了不起的程序员；我只是一个拥有良好习惯的好程序员。”<br>
> — **Kent Beck**

> “给一个已经延期的软件项目增加人手，只会让它更晚完成。”<br>
> — **Fred Brooks**

> “干净的代码简单而直接。干净的代码读起来像写得很好的散文。”<br>
> — **Robert C. Martin**

> “做那个可能有效的最简单的东西。”<br>
> — **Kent Beck**

> “你不是因为有对象才发送消息；你是因为发送消息，才需要对象。”<br>
> — **Sandi Metz**

> “你的应用现在只需要正确工作一次；但它必须永远易于变更。”<br>
> — **Sandi Metz**

> “现代软件实践与建筑或施工并不太相似。建筑大多已经建好了。如今，我们更多是在让一个预制好的空间适合居住其中的人。”<br>
> — **Sarah Mei**

## 核心原则

### 0. 宜居代码（Livable Code）

**描述**：软件应为真正需要长期生活在其中的人而设计，而不仅仅是为了架构纯粹性或技术上的优雅。

**理由**：开发者大多数时间都在与现有代码打交道。代码库应当是一个长期舒适、易于维护、易于演进的工作环境。

**智慧**：
> “你的代码，是像真人秀里那种杂乱无章的房子？还是像建筑杂志里那种光鲜极简的房子？这两种听起来都不像是你能舒服住进去的地方。”<br>
> — **Sarah Mei**

**应用**：
- 为开发者日常在代码库中的工作体验而设计
- 在技术纯粹性与实际需求之间取得平衡
- 考虑理解和导航系统时的认知负担
- 让代码库对新加入的人足够友好
- 当 code smell 持续出现时，去调试沟通模式

### 1. 清晰优于炫技（Clarity over Cleverness）

**描述**：优先选择简单清晰的设计，而不是聪明但复杂的方案，即使后者看起来更优雅或略高效。

**理由**：清晰的设计更易理解、维护、扩展和调试，也能降低认知负担，让新开发者更快上手。

**智慧**：
> “我把架构定义为：当我们本来想谈设计，但又想把它说得更重要时会用的一个词。”<br>
> — **Martin Fowler**

**应用**：
- 优先使用直白的实现，而非复杂抽象
- 在可能时优先采用成熟模式，而不是新奇做法
- 记录设计决策背后的 “why”，而不只是 “how”
- 如果一件事还需要费力解释它如何工作，那它大概率太复杂了

### 2. 关注点分离（Separation of Concerns）

**描述**：系统应划分为职责明确、功能重叠最小的不同部分。每个组件都应有清晰责任边界。

**理由**：良好分离的组件可以独立开发、测试和维护，从而降低认知负担并简化变更。

**智慧**：
> “单一职责原则。一个模块只应对一个，也仅对一个行动者负责。”<br>
> — **Robert C. Martin**

**应用**：
- 在子系统之间定义清晰边界
- 设计单一职责的组件
- 避免共享状态与隐式依赖
- 使用接口解耦实现细节
- 把领域模型视为系统核心

### 3. 可演化性（Evolvability）

**描述**：架构应促进系统随时间发生变化与演进，而不需要整体重写。

**理由**：需求总会变化，适应变化的能力是长期可持续性的关键。

**智慧**：
> “架构代表那些塑造系统的重要设计决策，而‘重要’是以变更成本来衡量的。”<br>
> — **Grady Booch**

> “未来是不确定的，而你永远不会比现在知道得更少。”<br>
> — **Sandi Metz**

**应用**：
- 在关键变化点上为可扩展性做设计
- 在组件之间建立稳定接口
- 使用抽象层隔离变更
- 预判并规划版本切换
- 建立能帮助团队对抗复杂性的模型
- 抵制为想象中的未来需求而过度设计的冲动

### 4. 可观测性（Observability）

**描述**：系统应能提供其行为、性能与状态的可见性。

**理由**：高效的监控、调试与性能优化都依赖于对系统运行状况的可见性。

**智慧**：
> “真相只能在一个地方找到：代码里。”<br>
> — **Robert C. Martin**

**应用**：
- 从一开始就内建日志、指标和追踪
- 让错误状态和边界情况显式且可观测
- 设计能够暴露健康状态和性能数据的组件
- 为关键操作建立清晰的审计轨迹
- 倾向显式行为，而不是隐式行为

### 5. 设计即安全（Security by Design）

**描述**：安全必须是架构的内建组成部分，而不是事后补丁。

**理由**：事后补安全更昂贵、效果更差，而且常常仍会留下漏洞。

**智慧**：
> “没有测试的代码就不是干净的代码。不管它多优雅、多易读、多易于访问，只要没有测试，它就是不干净的。”<br>
> — **Robert C. Martin**

**应用**：
- 在各个层面实现正确的认证与授权
- 应用最小权限原则
- 采用安全默认值
- 在设计阶段进行威胁建模
- 校验所有输入并净化所有输出

### 6. 领域中心设计（Domain-Centric Design）

**描述**：架构应反映并服务于问题领域，使用统一语言连接技术关注点与业务关注点。

**理由**：当模型与实现紧密贴合业务领域时，系统更可能满足用户需求，也更能适应需求变化。

**智慧**：
> “模型是一种经过有意识结构化和选择性简化的知识形式。”<br>
> — **Eric Evans**

**应用**：
- 建立开发者与领域专家共享的统一语言
- 建模领域，而不只是建模数据
- 将核心领域与支撑子域分离
- 让隐含概念在模型中显式化
- 随着理解加深持续打磨模型

### 7. 务实的简洁性（Pragmatic Simplicity）

**描述**：优先选择实用、能工作的方案，而不是理论上的完美。要认识到软件架构是在管理复杂性，而不是消灭复杂性。

**理由**：理论纯粹性往往会带来额外复杂度，却未必能带来成比例收益。真实系统必须平衡许多彼此竞争的关注点。

**智慧**：
> “理论和实践有时会冲突。而一旦冲突发生，输的永远是理论。”<br>
> — **Linus Torvalds**

> “与其先决定一个类，再思考它的职责；不如先决定一条消息，再思考它应该发送给谁。从以类为中心的设计转向以消息为中心的设计，是你设计生涯中的一个转折点。”<br>
> — **Sandi Metz**

**应用**：
- 重视可工作的软件，而不是面面俱到的文档
- 通过渐进改进，而不是追求完美，来推动演进
- 质疑那些无法明确简化系统的抽象
- 将运维关注点与架构纯粹性一起考虑
- 必要时接受务实的取舍
- 从消息向外设计，而不是从类向内设计
- 当“不做”的未来成本与“现在做”的当前成本相同，就推迟决策

### 8. 变更影响意识（Change Impact Awareness）

**描述**：每项架构决策在实施前，都应显式考虑它的影响范围、可逆性、时机与社会成本。

**理由**：资深工程师通常会本能地评估这些因素，但这份“无声清单”很少被记录。把变更影响显式化，有助于做出更好的决策并共享知识。

**智慧**：
> “解释这个行为本身很重要。它会让你放慢速度，从而注意到自己的直觉何时失准。它也会让你的推理变得可见，而这往往是单独工作时做不到的。”<br>
> — **Obie Fernandez**


**应用**：
- **影响范围**：识别变更失败或需要回滚时的影响范围。哪些组件、团队和用户会受到影响？
- **可逆性**：尽可能让变更可逆。问自己：“如果我们错了，回退有多难？”优先保留选择空间的方案。
- **顺序与时机**：考虑系统和团队是否已准备好接受变更。技术上正确的变更，时机不对也可能是错误的。
- **社会成本**：评估方案是否会让更多人困惑而不是受益。考虑团队的学习成本与认知负担。
- **虚假信心**：质疑“测试通过”是否真的能证明模型正确。寻找那些“实现正确”并不代表“方案正确”的地方。
- **变更定性**：澄清你是在引入新想法，还是在新的位置表达已有想法。理解变更应停留在表层还是深入系统内部。
- **扩散分析**：如果这种模式或方法会在整个代码库中扩散，这是好事还是负担？

**示例**：

*好的示例 - 高变更影响意识：*
```
决策：引入新的认证模式
影响范围：影响 23 个服务、4 个团队，需要协调
可逆性：中等 - 可回滚，但需要迁移脚本
时机：系统已准备好（可观测性已到位），但 Team B 需要培训
社会成本：学习曲线较高，但可提升长期可维护性
决策：先对 Team B 培训，再分阶段上线
```

*不好的示例 - 忽视影响：*
```
决策：引入新的认证模式
实现方式：在一个版本中改动所有服务
结果：所有服务全面故障，没有回滚计划，团队也很困惑
```

## 原则的演进

这些原则应在以下时机进行评审并视情况修订：
- 每个主要版本开发周期开始时
- 当持续遇到这些原则不足以指导决策的情况时
- 当新团队成员带来有价值的替代视角时

原则的拟议变更应由整个架构团队讨论，并在形成共识后采纳。

> “我坚持认为，在系统设计中，概念完整性是最重要的考量。”<br>
> — **Fred Brooks**

> “成功（能运行）但未经设计的应用，天然携带着自我毁灭的种子；它们很容易写出来，却会逐渐变得几乎无法修改。”<br>
> — **Sandi Metz**

> “通过反思已经写出的代码，以及不断出现的 code smells，你可以发现沟通中的缺失。你当然可以继续处理这些 code smells，甚至每个 sprint 都重构掉一次！但那只是在解决症状。更深层的问题更可能是沟通本身。”<br>
> — **Sarah Mei**
