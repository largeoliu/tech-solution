# 步骤 3：定制架构原则

## 输入
- 步骤 1 的项目上下文与 `project_signals`（状态文件 checkpoints.step-1）
- templates/principles-template.md

## 操作

### 生成流程
1. 从模板 `templates/principles-template.md` 定制生成 `.architecture/principles.md`
2. 按七个主章节填写项目事实、边界、底线和当前现实，必要时引用步骤 1 的 `project_signals`
3. 模板正文按七个主章节组织；如果模板定义了附录 `原则映射表`，则附录应一并保留
4. 遍历模板七个主章节，按证据情况分三种处理：
   - **有项目依据**：生成完整章节内容，`本项目事实` 填写项目事实
   - **无项目依据，也无明确反证**：生成章节框架，`本项目事实` 标注"暂未发现项目特异性依据"，保留模板的 `为什么重要` 和 `对方案与评审的要求`（评审框架，非编造约束）
   - **有明确反证**（如纯 CLI 工具无数据库，"数据接口与事件边界"确实不适用）：跳过，在该章节位置写一行 `[本章节经判定与当前项目无关，不生成原则内容。]`；附录 `原则映射表` 不计入七个主章节统计
5. 记录汇总：模板 7 个主章节中 M 个有依据生成 / F 个框架生成（无依据但无反证）/ X 个跳过（有明确反证）

### 七个主章节
模板定义的七个主章节为：
1. 文档目的与使用方式
2. 业务域与项目语义
3. 模块边界与依赖方向
4. 数据接口与事件边界
5. 质量属性与治理底线
6. 当前必须尊重的项目现实
7. 方案编写与评审准绳

### 填写规则
- 核心原则必须保留，但必须翻译成项目语境下的判断规则
- 有依据则生成对应章节内容；无依据但无反证则生成章节框架（`本项目事实` 标注"暂未发现项目特异性依据"）；有明确反证才跳过并写一行标注
- `.architecture/principles.md` 只保留对后续方案/评审有用的原则内容，不包含"依据"字段
- 将章节与依据编号的映射记录到状态文件 `checkpoints.step-3`，不在 `principles.md` 中显示
- 如果模板定义了附录 `原则映射表`，应将其视为附录保留，而不是额外的第八个主章节

### 输出边界约束
- principles.md 正文只包含模板定义的七个主章节
- 可附带模板定义的附录 `原则映射表`
- 不包含"文档维护"等模板之外的章节
- 跳过的章节保留标题和一行 `[本章节经判定与当前项目无关，不生成原则内容。]`，不直接删除章节标题，以保持七个主章节的结构可见

## 完成标准
- .architecture/principles.md 存在
- 可作为后续技术方案编写和架构评审的项目决策基线
- 章节与依据映射已记录到状态文件 `checkpoints.step-3.yaml`
- 来源汇总已记录（M 个章节有依据生成 / F 个章节框架生成 / X 个章节有反证跳过）

## 输出
- 更新状态文件 checkpoints.step-3.yaml
- 写入 .architecture/principles.md

## 门控
项目上下文或 `project_signals` 不足以进行原则定制时返回 STOP_AND_ASK；步骤 3 必须在步骤 4 之前完成
