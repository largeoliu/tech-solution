# 技术方案弱模型质量提升设计

**日期**：2026-03-27

**目标**：重构 `create-technical-solution` 的内部推理 contract，让同一个较弱模型在不依赖更强模型兜底的前提下，也能更稳定地产出结构完整、分析不跑偏、严格服从当前模板的技术方案文档。

## 问题与目标

当前 `create-technical-solution` 的正式规则已经比较完整，但它对强模型更友好：模板适配、方案分类、成员选择、专家分析、协作收敛和最终成稿被一条长链串在一起，且约束分散在 `SKILL.md` 和多个 references 中。强模型通常能自己补齐隐含顺序，较弱模型则更容易出现以下问题：

- 先按默认结构思考，再回头勉强适配当前模板。
- 专家产物使用通用标题，但和用户自定义模板语义脱节。
- 中间产物写得很满，但最终落回模板时遗漏槽位、引入空话或互相矛盾。
- 当模板语义不足或没有合法落点时，继续脑补内容而不是停机提问。

本设计的目标不是增加更多润色步骤，而是把内部推理过程改成更窄、更稳定、更容易验证的链路，让弱模型从一开始就只围绕当前模板已有槽位工作。

## 设计原则

- 当前 `.architecture/templates/technical-solution-template.md` 不只是最终正文骨架，而是整条分析链唯一的信息架构。
- 从专家分析到协作收敛再到最终成稿，任何阶段都不能引入模板外的组织层级。
- 最终技术方案文档必须严格填写当前模板已有位置，不能新增 section、subsection、appendix、独立表块或其他可见内容。
- 如果某个必要信息没有模板内合法落点，skill 必须停止并向用户提问，不能靠默认章节或补充段落解决。
- 模板必须先于方案类型、成员选择和专家任务设计被理解；后续所有步骤都消费模板语义结果，而不是重新自由解释模板。
- 中间产物必须短、窄、固定，优先降低弱模型的认知负担，而不是追求一次性写出大段完整分析。
- 对外使用方式尽量保持不变：前置文件检查、输出路径、覆盖确认、模板优先顺序等硬规则继续保留。

## 非目标

- 不修改 `.architecture/technical-solutions/[slug].md` 的正式输出位置。
- 不放宽现有前置文件要求，也不在缺前置时自动补环境。
- 不引入更强模型作为默认兜底策略。
- 不把中间产物落成新的侧车文档。
- 不通过新增“标准章节”解决自定义模板表达不足的问题。

## 新流程

`create-technical-solution` 的内部主链路调整为：

1. `范围冻结`
2. `模板任务单`
3. `专家按模板逐槽位分析`
4. `按模板逐槽位协作收敛`
5. `严格模板成稿`

这条链路的核心变化是：不再让模型先发明一个通用分析结构，再尝试把结果映射回模板；而是先把模板拆成可执行任务，再让每位专家只对模板已有槽位供给内容，最后仅把已经收敛的槽位内容填回模板。

## 中间产物 Contract

### 1. 范围冻结

`范围冻结` 只负责锁定问题边界，不负责发明正文结构。输出字段固定为：

- `问题`
- `目标`
- `非目标`
- `关键约束`
- `依赖`
- `影响范围`
- `slug`

只要这些字段发生变化，后续阶段都需要重新评估是否失效。

### 2. 模板任务单

`模板任务单` 是整条链路的核心产物。它必须严格按照当前模板的原始顺序展开每个已有槽位，并为每个槽位定义最小任务信息。推荐字段如下：

- `槽位标识`：来自当前模板的原始标题、表格位置或已有占位区域。
- `槽位顺序`：与当前模板一致，禁止重排。
- `槽位语义`：这个槽位在当前模板里承载什么内容。
- `可填写边界`：允许写入什么，不允许写入什么。
- `参与专家`：哪些专家需要参与这个槽位。
- `每位专家必答问题`：每位参与专家在该槽位必须回答的问题。
- `禁止越界项`：该槽位不能偷偷承担的内容。
- `依赖槽位`：和其他槽位的前后依赖或一致性关系。
- `缺口或阻塞项`：当前无法合法填写、需要停机补问的信息。

用户提出的“专家罗盘”不再作为单独的模板外层级存在，而是内嵌到 `模板任务单` 中：每个槽位都显式写出各专家在该槽位的职责和必答问题，从而把专家视角直接绑定到模板结构上。

### 3. 专家按模板逐槽位分析

每位专家只接收 `范围冻结` 和 `模板任务单`，并且必须按模板槽位顺序输出。专家不能再以 `设计目标`、`关键约束`、`主要风险`、`关键权衡` 等通用标题组织自己的主结构，而只能围绕模板已有槽位回答。

每个槽位下，专家的最小输出应包含：

- `是否参与该槽位`
- `建议写法或建议内容`
- `支撑理由`
- `与其他槽位的依赖或冲突`
- `仍需补证的信息`

如果某个槽位与该专家无关，应明确标记无贡献，而不是自由发挥到别的结构里。

### 4. 按模板逐槽位协作收敛

协作收敛阶段仍然是 process artifact，但它也必须严格围绕模板槽位组织，不能退回模板外的统一讨论纪要结构。每个槽位只收敛与该槽位直接相关的结论，例如：

- `共同结论`
- `分歧点`
- `选定写法`
- `未采纳理由`
- `仍待决策`

这些字段是每个槽位下的最小协作元数据，而不是新的模板章节。若某项在当前槽位语义下不成立，应显式留空或标记不适用，而不是硬塞内容。

### 5. 严格模板成稿

最终成稿阶段不再负责“重写一篇技术方案”，而只负责把已经收敛的稳定结论填回当前模板已有位置。它必须遵守三条硬规则：

- 只写入模板已有槽位。
- 只写入上游已收敛的内容，不新增新的架构判断。
- 遇到无合法落点的信息时立即停机提问。

因此，最终阶段更像“模板填充器”，而不是自由写作器。

## 失效与回退规则

- 当前模板发生变化时，至少回退到 `模板任务单`；既有专家分析和协作收敛全部失效。
- `问题`、`目标`、`非目标`、`关键约束` 或 `影响范围` 变化时，默认从 `范围冻结` 之后整条链路重算。
- 新增或替换专家成员配置时，至少回退到 `模板任务单`，重新分配槽位职责。
- 如果在任一阶段发现模板语义不足以支撑必要信息，必须立刻停止并提问，不能继续向后传播不合法的假设。
- 若用户在看到中间产物后新增约束，系统应明确说明哪些槽位失效、从哪个阶段重新开始，而不是沿用旧结论。

- 将主流程描述改为新五阶段链路。
- 移除任何暗示“专家先按统一 schema 思考、最后再适配模板”的描述。

### `skills/create-technical-solution/references/solution-process.md`

- 这是本次改动的核心文件。
- 将当前偏模板无关的专家 schema 改为“模板逐槽位 schema”。
- 定义 `模板任务单`、`专家逐槽位产物`、`逐槽位协作收敛` 的输入、输出和最小字段集。
- 明确通用分析概念只能作为槽位内的理由或判断依据存在，不能升格为模板外主结构。

### `skills/create-technical-solution/references/template-adaptation.md`

- 强化规则：模板不只是最终正文骨架，而是整条分析链唯一的信息架构。
- 明确任一阶段都不能长出模板外的章节层级。
- 明确若模板无法安全承载必要信息，必须停机提问，而不是回退默认结构。

### `tests/skill_validation/test_create_technical_solution_contracts.py`

- 补充新 contract 的静态断言和流程断言。
- 锁定“模板优先、逐槽位分析、最终零新增内容、无合法落点即停机”四类关键行为。

## 验证策略

验证重点不放在措辞优美，而放在结构服从与停机行为。至少补齐以下断言：

- 先理解模板语义，再设计专家任务，不能跳过 `模板任务单` 思维层。
- 专家和协作产物只能引用模板已有槽位，且顺序与模板一致。
- 在自定义模板下，不能再使用 `设计目标`、`主要风险`、`关键权衡` 等通用标题作为专家主结构。
- 最终技术方案正文不能新增 section、subsection、appendix、独立表块或其他模板外可见内容。
- 某项必要信息没有模板内合法落点时，结果必须是 `STOP_AND_ASK`，而不是偷偷补一段。
- 模板发生变化时，至少回退到 `模板任务单`，不能继续复用旧的专家分析。

建议最少新增三组回归场景：

- `默认模板`：验证标准主链路不回退。
- `异形自定义模板`：验证专家分析是否真正被自定义章节驱动。
- `语义过稀或不清模板`：验证技能是否正确停机，而不是脑补结构。

## 主要风险与边界

- `模板语义过弱`：只有标题没有含义时，较弱模型仍可能误判槽位职责，因此必须依赖明确停机规则。
- `必要信息无槽位`：若模板本身不承载候选方案、风险或未决事项，skill 不能替用户扩模板，只能停机提问。
- `逐槽位分析过碎`：结构锁紧后可能让内容显得碎片化，因此 `模板任务单` 必须提前声明跨槽位依赖，降低各写各的风险。
- `旧验证口径漂移`：如果测试仍按旧的统一专家 schema 断言，会误导后续修改，因此验证需要先更新。

## 实施顺序

1. 先修改 `skills/create-technical-solution/references/solution-process.md`，定下新的中间产物 contract。
2. 再修改 `skills/create-technical-solution/references/template-adaptation.md` 与 `skills/create-technical-solution/references/progress-transparency.md`，补齐模板唯一信息架构与回退规则。
3. 最后修改 `skills/create-technical-solution/SKILL.md`，让主 skill 只负责编排新链路。
4. 在 `tests/skill_validation/test_create_technical_solution_contracts.py` 中补齐回归断言，锁死新 contract。

这个顺序的目的，是先固定中间产物，再固定主流程，最后用测试收口，避免文档、主 skill 和验证三边互相打架。

## 预期结果

完成本设计后，`create-technical-solution` 对外入口基本不变，但内部会从“通用分析再适配模板”的强模型友好流程，收敛为“模板先行、逐槽位供给、严格原位成稿”的弱模型友好流程。即使模型写作能力一般，只要它能按照固定槽位逐步完成任务，最终技术方案的结构完整性、模板服从度和分析稳定性也会显著提升。
