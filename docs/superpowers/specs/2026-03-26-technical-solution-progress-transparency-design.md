# 技术方案阶段性进展透明化设计

**日期**：2026-03-26

**目标**：在 `create-technical-solution` 编写正式技术方案时，按阶段向用户展示结构化中间产物，降低长时间等待的黑盒感，让用户能在最终成稿前更早纠偏。

## 范围

- 展示每位专家的阶段性文档产物。
- 展示专家之间的协作收敛讨论结果。
- 中间产物默认仅在对话中展示，不新增侧车文档。
- 讨论过程默认展示为结构化纪要，而不是原始对话。

## 非范围

- 不做完整工作台式的进度中心、状态面板或事件流系统。
- 不改变最终技术方案文档在 `.architecture/technical-solutions/` 中的正式产物角色。
- 不暴露内部 scratchpad、原始推理链路或未成形草稿。

## 架构

在 `create-technical-solution` 现有主流程 `专家独立输入 -> 协作收敛 -> 最终成稿` 之上增加一层“阶段性播报协议”。正式技术方案仍然遵循当前模板落位，不改为新的正文结构；新增协议只负责说明在关键阶段如何把稳定的中间产物展示给用户。

专家阶段沿用 `solution-process.md` 中既有 schema，以结构化专家卡片方式展示：`视角说明`、`设计目标`、`关键约束`、`推荐做法`、`主要风险`、`关键权衡`、`未决问题`。协作阶段展示结构化讨论纪要，覆盖共同结论、争议点、候选方案对比、选定方向、选择理由、原则冲突与取舍、未决问题与后续决策点。

## 组件

### 阶段控制器

- 识别当前处于专家独立输入、协作收敛还是最终成稿阶段。
- 在阶段边界触发对话内播报。
- 在用户中途补充约束时，决定从哪个最近受影响的阶段边界恢复。

### 专家产物渲染器

- 将单个专家的独立输入渲染成结构化专家卡片。
- 仅在必填字段补齐后展示。
- 长内容采用“摘要优先，详细展开”的呈现方式。

### 讨论纪要渲染器

- 将全部专家输入收敛为结构化协作纪要。
- 保留争议点、被否决方案及未选原因。
- 不输出原始争论聊天记录。

### 最终方案装配器

- 将选定方向和权衡映射回当前技术方案模板。
- 保证模板仍是唯一正文骨架。
- 不因中间产物展示而引入新的正式章节体系。

### 展示边界规则

- 只展示稳定、可解释、可归因的结果。
- 不展示内部不稳定内容。
- 当用户修改前提时，先解释失效范围，再进入重算。

## 数据流

1. 保持现有准备阶段不变：主题、模板、参与成员、共享上下文。
2. 在“专家独立输入”阶段，每位专家完成后立即展示一张结构化专家卡片。
3. 在全部专家输入完成后，先生成并展示结构化协作收敛纪要。
4. 用户看过纪要后，再将选定方向、权衡、风险等内容映射到最终模板并成稿。
5. 如果用户在专家卡片或协作纪要后介入，系统从最近受影响的阶段边界恢复，而不是整条链路全部重来。

## 错误处理与回退

- 字段未补齐前，不展示不完整专家卡片。
- 若专家意见强冲突，协作纪要必须显式保留争议与被否决方向。
- 若内容过长，先给摘要，再允许查看详细内容。
- 若用户新增约束或修正目标，必须明确说明哪些既有结论失效、从哪个阶段重新开始。
- 不允许把旧结论继续表述为当前有效结论。

## 验证与验收

- 最终技术方案文档生成行为不回归。
- 常规场景下，用户能尽快看到第一份专家产物。
- 强分歧场景下，用户能在协作纪要中看到争议点和被否决方案。
- 用户中途插话场景下，系统能明确输出失效范围与重算起点。
- 展示边界安全：不泄漏 scratchpad、原始推理或不稳定草稿。

## 实现思路

- 新增独立参考文档定义阶段性播报协议，保持职责清晰。
- 在 `SKILL.md` 中只补充完成标准、工作流接线、行为契约和结果汇报。
- 在 `solution-process.md` 中把既有 schema 映射为用户可见产物，并补充阶段回退规则与质量门槛。
- 在 `README.md` 中更新使用预期，让用户知道正式文档落盘前会先看到结构化中间产物。
