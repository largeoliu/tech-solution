# 宿主项目原则文档设计

**日期**：2026-03-27

**目标**：重构 `setup-architect` 产出的 `.architecture/principles.md` 设计方式，使其优先提供宿主项目上下文与判断基线，直接支撑后续技术方案编写与评审，而不是停留在通用架构格言或需要用户自行拼装的原则清单。

## 范围

- 调整 `skills/setup-architect/templates/principles-template.md`，让默认模板直接体现“宿主项目判断基线”结构。
- 调整 `skills/setup-architect/references/principles-customization.md`，明确指导用户如何把宿主项目事实与抽象原则整合进 `.architecture/principles.md`。
- 轻量补强 `skills/setup-architect/SKILL.md` 中“定制架构原则”的定位说明，使其与新的模板和指导文档一致。
- 增加或收紧 `tests/skill_validation/` 下与 `setup-architect` 相关的静态契约，防止原则模板和说明文档漂移回“通用原则清单”模式。

## 非范围

- 不修改 `create-technical-solution` 的主流程、模板链路或输出结构。
- 不修改 `review-technical-solution` 的正式评审流程、输出契约或证据规则。
- 不为任意宿主项目自动生成真实业务内容；模板仍然只提供结构、问题和约束骨架。
- 不把 `.architecture/principles.md` 扩展为 README、系统清单、接口总表或运维手册。
- 不要求删除现有核心原则；核心原则仍必须保留，只改变它们的表达和落位方式。

## 定位

`.architecture/principles.md` 的主要作用，是为宿主项目后续的技术方案 skill 提供可复用的项目上下文和判断基线。它既不是抽象口号集合，也不是完整现状快照，而是只保留那些会影响方案选型、边界判断、风险识别和评审结论的项目事实与规则。

文档应同时承载两类信息：第一类是相对稳定的长期基线，例如业务域划分、依赖方向、接口边界、质量底线和决策标准；第二类是必须被当前方案尊重的现实约束，例如遗留模块、迁移阶段、历史兼容包袱、组织协作边界或短期不可突破的禁区。两者缺一不可：只有稳定基线会失去现实约束，只有现状快照又会失去长期判断标准。

## 推荐结构

推荐把 `.architecture/principles.md` 固定为以下七个主章节，并在文末增加一个简短的 `原则映射表`：

1. `文档目的与使用方式`
2. `业务域与项目语义`
3. `模块边界与依赖方向`
4. `数据、接口与事件边界`
5. `质量属性与治理底线`
6. `当前必须尊重的项目现实`
7. `方案编写与评审准绳`

这个结构优先服务于后续方案写作和评审，而不是服务于“原则展示”。前六章负责沉淀宿主项目的上下文、边界与底线，第七章负责把这些信息收束成可执行的判断规则。文末 `原则映射表` 只承担追溯和复核作用，不应成为正文主结构。

## 章节写法

每个章节都应回答三个问题：这个项目里“它是什么”、为什么它会影响技术方案和评审、它对后续方案和评审意味着什么。推荐每章都按以下三层组织：

- `本项目事实`：写清宿主项目的稳定语义、边界、底线或阶段性现实。
- `为什么重要`：解释这些事实会约束哪些方案选择，避免什么风险。
- `对方案与评审的要求`：给出明确的判断句、禁区句和优先级规则。

正文语言应尽量使用规则句，例如 `必须`、`优先`、`禁止`、`仅在……时允许`。除非宿主项目确实需要弹性，否则避免使用 `通常`、`尽量`、`一般情况下` 这类容易让评审尺度漂移的措辞。

## 内容粒度规则

`.architecture/principles.md` 只应纳入会改变后续方案选择、边界判断或评审结果的内容。推荐保留：

- 稳定业务域划分和关键术语定义。
- 重要模块职责、依赖方向和跨模块协作约束。
- 关键数据、接口、事件边界及兼容性约束。
- 测试、发布、回滚、观测、安全、合规等治理底线。
- 当前必须尊重的遗留现实、迁移阶段和组织边界。
- 技术方案与实施计划的决策标准。

推荐排除：

- README 式项目介绍和泛泛背景说明。
- 全量系统、接口、表结构或配置清单。
- 详细运维操作步骤。
- 高频变动的临时事实。
- 冗长 ADR 历史或不再影响当前判断的旧决策过程。

## 核心原则的融合方式

保留现有九条核心原则，但不再把它们主要作为独立、悬空的通用章节保留。新的推荐写法是“映射式融合”：在每个宿主项目章节中同时写清项目事实、对应原则和由此导出的要求，让原则变成“这在当前项目里意味着什么”，而不是停留在抽象格言。

推荐映射如下：

- `宜居代码`：落在 `文档目的与使用方式`、`模块边界与依赖方向`。
- `清晰优于炫技`：落在 `模块边界与依赖方向`、`方案编写与评审准绳`。
- `关注点分离`：落在 `模块边界与依赖方向`、`数据、接口与事件边界`。
- `可演化性`：落在 `数据、接口与事件边界`、`质量属性与治理底线`。
- `可观测性`：落在 `质量属性与治理底线`。
- `设计即安全`：落在 `质量属性与治理底线`。
- `领域中心设计`：落在 `业务域与项目语义`。
- `务实的简洁性`：落在 `当前必须尊重的项目现实`。
- `变更影响意识`：落在 `当前必须尊重的项目现实`、`方案编写与评审准绳`。

文末 `原则映射表` 只需要简要列出“原则 -> 章节 -> 在本项目中的约束含义”，方便人类复核和 skill 追溯。

## 模板设计要求

新的 `principles-template.md` 应直接提供上述七章骨架，并在每章中包含可以填写的提示项，帮助宿主项目维护者写出对后续技术方案真正有用的内容。模板不应再把“项目特定原则”主要放在文末 HTML 注释中引导用户自行复制拼装，而应把项目特定内容提升为正文主结构。

模板仍需保留九条核心原则的名称和核心含义，但呈现方式要改成与宿主项目章节联动的写法。可接受的方式包括：在章节中内嵌“相关原则”提示，以及在文末提供简短映射表；不接受的方式是重新退回到“前半部分是通用格言，后半部分才是项目补充”的主次结构。

## 引导文档要求

`principles-customization.md` 应把当前的编辑指导改成“按章节填写宿主项目判断基线”，而不是“从 HTML 注释复制一个项目特定原则小节”。新的引导文档至少需要明确以下规则：

- 核心原则必须保留，但必须翻译成宿主项目语境下的判断规则。
- 内容必须同时覆盖稳定基线和最小必要的当前现实。
- 只写会影响技术方案和评审结果的内容，避免写成百科式文档。
- 至少覆盖模块边界与依赖方向、API / 事件 / 数据边界、测试和验证基线、安全与合规底线，以及技术方案和实施计划的决策标准。
- 每章都应尽量给出可执行、可评审、可争议复核的判断句。

## Skill 与验证联动

`setup-architect` 主 skill 不需要改变流程顺序，仍保持“安装 -> 分析项目 -> 定制成员 -> 定制原则 -> 复核结构 -> 确认模板”的主路径。需要补强的是第 4 步的语义定位，让 skill 明确告知用户：原则文档不是可选背景，而是后续技术方案与评审的必需判断基线。

验证侧至少应补上两类收口：

- `test_setup_architect_contracts.py` 要能读取 `principles-template.md`，并断言新模板包含七个主章节与原则映射结构。
- `test_setup_architect_contracts.py` 或 `case_catalog.py` 要断言 `principles-customization.md` 明确要求宿主项目上下文、核心原则保留、最低覆盖范围、以及“只保留影响方案与评审判断的信息”。

## 验收标准

- 默认 `principles-template.md` 能直接作为宿主项目 `.architecture/principles.md` 的起点，不需要先从注释区拼装正文结构。
- 新模板显式包含七个主章节和文末 `原则映射表`。
- 新模板和引导文档都清楚表达：`.architecture/principles.md` 的主要用途是支撑技术方案编写与评审，而不是展示抽象原则。
- 九条核心原则仍然保留，但都能在宿主项目语境中找到明确落点。
- 合同测试能稳定阻止模板回退成“通用原则正文 + 项目补充注释”的旧模式。
- 现有 `create-technical-solution` 与 `review-technical-solution` 的前置依赖和主流程不被破坏。

## 实现思路

- 先通过测试把新结构与新语义锁住，再修改模板和说明文档，保持测试先行。
- 对 `setup-architect/SKILL.md` 只做最小必要变更，避免影响已稳定的安装与模板替换契约。
- 保持仓库现有风格：主 skill 文档短而硬，细节规则下沉到 references，验证通过 Python `unittest` 合同测试完成。
