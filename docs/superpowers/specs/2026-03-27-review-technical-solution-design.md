# 技术方案评审 Skill 设计

**日期**：2026-03-27

**目标**：新增 `review-technical-solution` skill，使其在用户提供需求详情、技术方案文档、`.architecture/principles.md` 与相关项目代码时，能够基于证据输出结构化评审报告，指出方案问题、说明影响，并给出可执行的改进建议。

## 范围

- 新增独立 skill：`skills/review-technical-solution/SKILL.md`。
- 新增三份配套参考文档：
  - `skills/review-technical-solution/references/review-process.md`
  - `skills/review-technical-solution/references/review-analysis-guide.md`
  - `skills/review-technical-solution/references/review-output-contract.md`
- 评审输入强制覆盖四类证据源：需求详情、技术方案文档、`.architecture/principles.md`、相关项目代码。
- 输出固定为结构化评审报告，包含结论、阻断项、主要问题、改进方案、待补充信息、建议验证。
- 为 skill validation 增加静态契约、流程场景、行为回归、对抗边界四层验证用例。

## 非范围

- 不负责自动生成完整技术方案正文；该职责仍属于 `create-technical-solution`。
- 不做“轻量随手点评”模式；缺少必要上下文时只允许返回 `无法开展正式评审`。
- 不直接改写用户提供的技术方案文档为正式成稿。
- 不引入新的架构原则来源；`.architecture/principles.md` 仍是正式原则基线。
- 不替代通用代码评审或 PR review；本 skill 的对象是“技术方案是否与现有架构和代码现实相符”。

## 架构

`review-technical-solution` 采用“主 Skill 负责边界与硬规则，references 负责流程、分类与输出契约”的组织方式，与现有 `create-technical-solution` 的结构保持一致。`SKILL.md` 保持短而硬，只定义何时使用、缺什么就不能正式评、必须检查哪些证据、最终输出必须长什么样。评审细节全部下沉到 references，避免主 skill 过长且便于契约测试逐项断言。

整个评审过程遵循“先校验输入，再提取主张，再做代码取证，再按维度归因，再输出问题与改进方案”的顺序。核心约束不是“尽量给建议”，而是“没有证据不下结论”。如果输入不完整、代码无法定位、架构原则缺失或方案正文不完整，skill 必须明确阻断并说明为什么当前无法给出可信的正式评审结果。

## 组件

### 主 Skill 入口

- 定义适用场景、非适用场景、必要上下文与硬性阻断规则。
- 强制要求评审同时读取需求、方案、原则文档和相关代码。
- 约束最终输出必须遵循固定区块与固定顺序。
- 收口红旗信号，防止在赶时间、证据不足或用户催促时降低评审标准。

### 通用评审流程参考

- 负责输入校验、方案分类、主张提取、代码取证、问题分级与最终结论判定。
- 定义 `已证实`、`已证伪`、`待核验` 三种证据状态。
- 定义 `通过`、`需修改`、`阻断`、`无法开展正式评审` 四种结论及其触发条件。
- 约束所有确认问题必须带证据，所有不确定问题必须降级为 `待核验风险`。

### 分类评审指引

- 负责根据方案类型调整检查重点。
- 覆盖五类主分类：新功能方案、重构或替换方案、跨系统或平台能力方案、安全 / 合规 / 数据保护方案、性能 / 容量 / 成本优化方案。
- 每个分类都固定输出：适用信号、必查问题、必查代码证据、常见阻断项、优先改进方向。
- 多分类命中时取并集，并按风险、爆炸半径和耦合度排序输出。

### 输出契约参考

- 定义正式输出的六个固定区块与严格顺序。
- 规定每类问题条目的最小字段集：`severity`、`category`、`problem`、`evidence`、`impact`、`recommendation`、`validation`。
- 规定所有区块都必须出现，空区块写 `- 无`。
- 规定 `改进方案` 必须显式关联问题编号，防止建议与问题脱钩。

### Skill Validation 套件

- 在 `tests/skill_validation/test_review_technical_solution_contracts.py` 中对 skill 文档和 references 做静态契约断言。
- 在 `tests/skill_validation/case_catalog.py` 中增加 `RTS-*` 场景集，覆盖缺失上下文、代码证据冲突、多分类方案、高压催促等场景。
- 在 `tests/skill_validation/helpers.py` 中加入 `review-technical-solution` 安装布局，确保测试可装载该 skill。

## 数据流

1. 用户提供需求详情与技术方案文档，请求进行技术方案评审。
2. Skill 首先确认是否具备四项必要输入：需求详情、方案正文、`.architecture/principles.md`、相关项目代码。
3. 若任一必要输入缺失，立即输出 `无法开展正式评审`，并在固定输出结构中列出阻断项和待补充信息。
4. 若输入齐备，skill 根据方案主要变化对象与最高风险来源确定主分类，并识别附加分类。
5. Skill 从需求与方案中提取核心主张，包括目标、约束、复用能力、变更边界、接口、数据结构、测试与发布策略。
6. Skill 在项目代码中核验这些主张是否有对应实现、边界是否匹配、依赖是否真实存在、现有运行时约束是否被方案覆盖。
7. Skill 按需求对齐、架构对齐、代码现状对齐、完整性、可落地性五个维度汇总问题。
8. Skill 根据证据强度与影响范围为问题分级，并生成对应的改进方案与建议验证动作。
9. Skill 按固定区块顺序输出正式评审报告。

## 错误处理与回退

- 缺少必要输入时，禁止输出 `通过`、`需修改` 或 `阻断`，只能输出 `无法开展正式评审`。
- 找不到方案声称复用的现有能力时，必须显式指出“代码证据缺失”或“主张被代码证伪”，不能用经验推断补齐。
- 如果只拿到片段代码且不足以判断核心边界，必须把相关结论降为未核验，而不是继续给确定性判断。
- 如果方案文本合理但代码证据不足，必须维持低置信度输出，不因文档表达质量降低检查强度。
- 如果用户要求跳过代码核验或快速给结论，skill 必须拒绝降低标准，并说明正式评审依赖代码证据。

## 验证与验收

- skill 文档中必须显式包含四类必要输入、硬性阻断规则、证据规则、固定输出区块、红旗信号。
- 缺少 `.architecture/principles.md`、缺少方案正文、缺少相关代码时，流程场景测试必须得到 `无法开展正式评审`。
- 当方案声称复用现有能力时，场景测试必须要求在代码中找到对应实现或边界证据。
- 多分类方案必须合并检查点，不能只按单一类别输出。
- 输出始终包含六个固定区块；存在 `blocker` 时，正式结论不得为 `通过` 或 `需修改`。
- `待核验风险` 必须同时包含已掌握证据、缺失证据和下一步核验动作。
- 对抗边界测试中，用户催促、只给摘要、只给片段代码、以“看起来合理”为由要求跳过核验时，skill 都不能退化为泛化点评。

## 实现思路

- 复用 `create-technical-solution` 的分类思路与风险维度，但不复用其生成正文职责。
- 让 `SKILL.md` 聚焦高强度规则，把大量细节沉入 references，减轻主 skill 体积并提升可维护性。
- 输出契约采用固定标题、固定字段、空区块显式化的方式，降低验证成本并提高稳定性。
- `description` 采用英文 `Use when ...` 风格，以兼容现有技能发现规则；正文继续使用中文，匹配项目现有技能文档风格。
- skill validation 按静态契约层、流程场景层、行为回归层、对抗边界层组织，保证这套评审能力既能被发现，也能在压力场景下守住证据底线。
