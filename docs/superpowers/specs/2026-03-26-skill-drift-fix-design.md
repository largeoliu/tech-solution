# Skill 漂移修复设计

## 概述

本设计用于修复 `setup-architect` 与 `create-technical-solution` 中已经识别出的 6 个流程漂移问题。

这次修复采用“结构性重写”思路：不是只在原文上零散补句子，而是重新梳理主 skill 与引用文档的职责边界，让主 skill 本身就能单独表达正确顺序、停机规则、完成标准和关键质量要求；引用文档继续保留细节、字段格式和补充规则，但不再承担“唯一能看到硬规则”的角色。

## 背景

当前两个 skill 的主要问题不是能力缺失，而是规则分布不均、顺序前后冲突、主文档过于依赖引用文档补全语义，导致执行者容易出现以下偏差：

- 只看主 skill 时漏掉模板替换硬规则。
- 同一 skill 内部出现两套步骤顺序。
- 不清楚输出目录缺失时应自动创建还是停机。
- 不深入引用文档时看不到关键质量要求和强制信息块。
- 顶层完成标准过宽，无法直接约束最终输出是否合格。

## 目标

- 让 `setup-architect` 的主文档就能清楚表达“完整初始化”和“仅替换模板”两条分支。
- 让 `create-technical-solution` 只有一条唯一的主执行路径，不再存在模板读取顺序和成员选择顺序冲突。
- 明确区分“语义前置”与“输出位置”，并把目录缺失行为写成显式规则。
- 把最关键的质量门槛和必填信息块摘要提升到主 skill。
- 收紧顶层完成标准，使其可以直接作为验收依据。
- 保持“技术方案模板可被替换”的能力，并始终围绕“当前生效模板”而不是默认模板展开。

## 非目标

- 不改变 `.architecture/templates/technical-solution-template.md` 的模板内容。
- 不取消模板替换能力。
- 不把引用文档全部搬回主 skill。
- 不把 `create-technical-solution` 改写成围绕默认模板章节名工作的技能。
- 不调整本次检查报告本身。

## 关键设计原则

### 1. 当前生效模板优先

`create-technical-solution` 永远围绕当前 `.architecture/templates/technical-solution-template.md` 工作。该模板可能是默认模板，也可能是用户在初始化阶段或初始化完成后单独替换过的自定义模板。

因此，主 skill 中提升的是“必须包含哪些语义信息”和“必须遵守哪些停机规则”，而不是把文档重新绑回默认模板结构。

### 2. 主 skill 是可独立执行的权威入口

执行者只看 `SKILL.md`，也必须能知道：

- 正确步骤顺序
- 哪些条件缺失时必须停止
- 哪些动作允许自动继续
- 哪些内容必须出现在最终产物中
- 什么状态才算完成

引用文档负责补细节，不负责单独定义关键停机点、唯一顺序和关键验收底线。

### 3. 主 skill 写摘要闭环，引用文档写完整版细则

主 skill 不追求把每一个字段说明全文复制过来，但必须给出“最小可执行闭环”。引用文档继续承接：

- 输入格式
- 信息块详细定义
- 模板适配细则
- 方案类型分析细则
- 场景化摘要文案

## 影响范围

预计涉及以下文件：

- `skills/setup-architect/SKILL.md`
- `skills/setup-architect/references/technical-solution-template-customization.md`
- `skills/create-technical-solution/SKILL.md`
- `skills/create-technical-solution/references/solution-process.md`
- `skills/create-technical-solution/references/solution-analysis-guide.md`
- `skills/create-technical-solution/references/template-adaptation.md`

## `setup-architect` 重构设计

### 主职责调整

`skills/setup-architect/SKILL.md` 将从“初始化主流程 + 文末补一句模板替换提示”改为“清楚分流的总入口文档”。

主文档开头直接区分两种入口：

- 完整初始化：创建 `.architecture/` 结构、安装基础文件、定制成员与原则、确认最终模板。
- setup 完成后仅替换技术方案模板：跳过初始化流程，只走模板替换分支。

### 模板替换规则上移

以下规则将直接写入 `skills/setup-architect/SKILL.md`，不再只藏在引用文档里：

- 替换模板前必须先做前置校验。
- 若 setup 不完整，则必须停止，并要求用户先完成完整初始化。
- 只接受完整 Markdown、文件路径或链接地址。
- 只允许整体替换 `.architecture/templates/technical-solution-template.md`。
- 不允许自动生成模板、局部编辑或内容合并。

### 完成标准调整

`setup-architect` 的完成标准会继续保留“模板可替换”的能力，并明确收敛为：

- `.architecture/` 最小结构已建立。
- `members.yml` 与 `principles.md` 已安装并按项目现实定制。
- `.architecture/templates/technical-solution-template.md` 已明确为当前生效模板。
- 当前生效模板可以是默认模板，也可以是用户替换后的自定义模板。

### 引用文档职责收缩

`skills/setup-architect/references/technical-solution-template-customization.md` 继续保留，但职责调整为：

- 提供模板替换输入细节。
- 提供初始化收尾场景与安装后替换场景的摘要文案。
- 解释通用输入、通用输出和具体场景差异。

它不再是“唯一定义模板替换硬规则的地方”。

## `create-technical-solution` 重构设计

### 唯一主路径

`skills/create-technical-solution/SKILL.md` 的高层工作流将重排为唯一顺序：

1. 定题与范围判断。
2. 检查语义前置文件。
3. 读取当前生效模板。
4. 判断方案类型。
5. 加载成员名册并选择参与者。
6. 构建共享上下文。
7. 组织成员独立输入。
8. 收敛为标准信息块。
9. 将信息块落位到当前模板并通过质量门槛。
10. 保存并汇报结果。

这条顺序用于统一解决两类确认漂移：

- “总是先读当前模板”与“第 4 步才读模板”的冲突。
- “先判方案类型再定成员”与“主流程先选成员”的冲突。

### 前置条件分层

主 skill 将明确区分两类前置：

#### 语义前置

以下文件缺失时，说明初始化未完成，必须立即停止并回指 `setup-architect`：

- `.architecture/members.yml`
- `.architecture/principles.md`
- `.architecture/templates/technical-solution-template.md`

#### 输出位置

`.architecture/technical-solutions/` 被定义为输出目录，而不是语义前置。若三个关键文件均存在，但该目录缺失，则由 `create-technical-solution` 自动创建该目录后继续执行。

这样可以避免把“缺少输出路径”和“缺少设计语义前提”混为一类。

### 当前模板语义澄清

主 skill 会统一使用“当前生效模板”这一表述，并显式说明：

- 当前模板可能是默认模板。
- 当前模板也可能是用户替换后的自定义模板。
- 方案生成时必须先读取它，再决定后续信息块如何落位。
- 不允许把自定义模板重写回默认模板结构。

### 主文档中的最小信息块摘要

`skills/create-technical-solution/SKILL.md` 会直接列出最关键的标准信息块摘要，至少包括：

- 问题与背景
- 目标与非目标
- 约束与依赖
- 推荐方案
- 备选方案与权衡
- 详细设计
- 风险与缓解
- 实施建议
- 评审关注点
- 未决问题

这里写的是“最小必须覆盖的语义内容”，而不是固定章节名。

如果当前生效模板没有与这些信息块一一对应的显式章节，skill 也不能因此跳过它们；它必须先基于模板语义，把这些内容落到现有章节、小节、表格或列表中。

如果模板结构过于自由，导致某个必需信息块无法在不破坏模板意图、且不发明新的顶级章节的前提下安全落位，skill 必须停止并向用户确认，而不是擅自补一套默认模板章节。

### 主文档中的最小质量门槛摘要

`skills/create-technical-solution/SKILL.md` 会提升出一组主文档可见的最小质量门槛，例如：

这些门槛始终建立在“当前生效模板优先”上：技术方案模板可以被替换，所以这里约束的是语义完整性、模板遵循和落位质量，而不是要求技能回退到默认模板章节结构。

- 每个标准信息块都已生成。
- 至少存在一个认真比较过的备选方案，并写明未选原因。
- 每个主要风险都包含影响、概率判断和缓解方向。
- 已明确边界与职责、依赖关系。
- 未决问题写明缺什么信息、由谁补齐。
- 最终内容已落到当前生效模板（无论它是默认模板还是已替换的自定义模板），而不是退回默认模板章节。

### 完成标准收紧

完成标准会从现在的概括式表述改为“摘要式闭环”。

新的完成标准要能直接约束：

- 文档已覆盖关键标准信息块。
- 参与成员选择与方案类型一致，且包含系统架构师。
- 已体现备选方案、未选原因、风险判断和缓解方向。
- 已体现边界与职责、依赖关系、实施建议和评审关注点。
- 已按当前生效模板完成落位。
- 保存行为符合覆盖确认规则。

## 引用文档职责重排

### `references/solution-analysis-guide.md`

继续只定义：

- 方案类型识别方法
- 不同类型的推荐成员
- 必答问题
- 易漏风险
- 评审重点

不再与主 skill 共同承担顶层顺序定义。

### `references/solution-process.md`

继续保留：

- 成员独立输入格式
- 协作收敛格式
- 10 类标准信息块的完整说明
- 完整质量门槛清单

但会与主 skill 对齐，让主 skill 提供“摘要闭环”，此文档提供“完整版细则”。

### `references/template-adaptation.md`

保持为模板落位细则文档，但会统一措辞，继续强调：

- 当前模板可能是默认模板，也可能是自定义模板。
- 必须以当前模板的实际结构为准。
- 不允许把用户模板改写回默认模板结构。

## 六个问题与修复映射

### 1. 模板替换硬规则藏在引用文档里

修复方式：把模板替换前置校验、失败即停止、整文件替换、禁止自动生成/局部编辑/合并等规则提升到 `skills/setup-architect/SKILL.md`。

### 2. “总是先读模板”与“后面才读模板”冲突

修复方式：重排 `create-technical-solution` 高层工作流，把“读取当前生效模板”固定到前置检查之后、类型分析之前。

### 3. “先分析类型再定成员”与“先选成员”冲突

修复方式：把类型分析前移到成员选择之前，并让分析指南只负责分析依据，不再暗中定义主流程顺序。

### 4. 输出目录不存在时行为不明确

修复方式：在主 skill 中显式区分语义前置与输出目录；关键文件齐全而目录缺失时，自动创建输出目录后继续。

### 5. 重要质量要求和必填内容藏在引用文档里

修复方式：把关键标准信息块和最小质量门槛摘要提升到 `create-technical-solution/SKILL.md`。

### 6. 顶层完成标准太宽泛

修复方式：把完成标准收紧为可直接验收的摘要闭环，明确要求关键信息块、风险判断、权衡、实施建议、评审关注点和模板落位都到位。

## 验收标准

修复完成后，应满足以下条件：

- 只看 `skills/setup-architect/SKILL.md`，也不会漏掉模板替换的关键停止条件和允许动作。
- `skills/create-technical-solution/SKILL.md` 中不再存在两套模板读取顺序。
- `skills/create-technical-solution/SKILL.md` 中不再存在两套“类型分析 / 成员选择”顺序。
- 三个关键语义前置缺失时会停止并回指 `setup-architect`。
- 仅输出目录缺失时会自动创建目录后继续。
- 主 skill 本身可见最关键的信息块和质量门槛摘要。
- 完成标准可以直接判断结果是否合格，而不只是抽象描述。
- 全部规则继续兼容“当前模板可为默认或自定义模板”。
- 最终生成的技术方案正文必须服从当前生效模板；如果模板没有同名章节，也只能按现有结构语义落位，不能脱离模板另起一套默认章节。
- 如果当前生效模板确实无法安全承载某个必需语义内容，skill 会停止并向用户确认，而不是绕开模板继续生成。

## 风险与控制

- 风险：主 skill 变长，可能产生与引用文档的重复。
  - 控制：主 skill 只写摘要闭环，引用文档保留完整版细节。

- 风险：在强调信息块时，误让执行者把它们理解成固定章节名。
  - 控制：全文统一强调“信息块是语义要求，不是默认模板章节清单”。

- 风险：调整顺序后，引用文档仍残留旧表达。
  - 控制：本次修复把相关引用文档一并纳入修改范围，统一措辞。
