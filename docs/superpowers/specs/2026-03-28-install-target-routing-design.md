# 安装目标路由设计

## 概述

本设计定义 skill 安装时的目标目录判定规则，目标是让安装行为基于“当前 AI 助手运行环境”稳定落到正确位置，而不是被项目里已经存在的 `.qoder/`、`.trae/`、`.claude/` 等目录误导。

核心原则只有两条：

- 路由证据只能来自助手宿主或运行时环境，不能来自项目目录特征。
- 能唯一判定时才安装；不能唯一判定时，必须失败并给出诊断，不能猜。

## 背景

当前错误模式是：安装器看到目标项目里存在某个助手目录，例如 `.qoder/`，就把这类目录当成“应该安装到这里”的信号。这样会导致在 Trae 里执行项目级安装时，即使当前运行环境是 Trae，只要项目里已有 `.qoder/`，安装结果也可能错误落到 `.qoder/skills`。

这个问题说明：

- 项目目录只能说明“这个仓库曾经被哪些助手使用过”，不能说明“当前是谁在执行安装”。
- 目录存在性推断会随着仓库历史不断积累噪音，时间越久越不可靠。
- 继续在安装逻辑里保留目录推断，只会把错误变成长期系统行为，而不是偶发问题。

因此，安装路由必须从“看项目里有什么目录”改成“先识别当前运行环境是谁，再走统一映射表”。

## 目标

- 让项目级安装在多助手混合仓库中仍然稳定命中正确目录。
- 为所有支持的助手使用统一路由协议，而不是为 Trae 单独加分支。
- 把 `assistant_id` 识别与目标目录映射拆成两个独立阶段，降低后续扩展成本。
- 在环境信号缺失、冲突或未知时明确失败，避免误装。
- 为后续增加新助手保留稳定扩展点：新增助手时只需要补运行时识别和映射表项。

## 非目标

- 本设计不把项目目录扫描保留为低优先级兜底。
- 本设计不要求今天所有助手都必须已经提供统一标准环境变量。
- 本设计不重写各助手自己的全局安装协议；它只要求全局目标也必须走同一套注册表解析，而不是目录猜测。
- 本设计不讨论 skill 内容本身，只讨论安装目标选择。

## 设计总览

安装目标解析拆成 4 个阶段：

1. 从当前宿主或运行时环境收集助手身份信号。
2. 把这些信号规范化为统一的 `assistant_id`。
3. 根据 `assistant_id + scope` 查询统一注册表得到目标路径。
4. 仅在结果唯一且合法时执行安装，否则失败并输出诊断。

可表示为：

```text
runtime signals -> normalized assistant_id -> TARGETS[assistant_id][scope] -> install
```

其中 `scope` 是独立输入，例如 `project` 或 `global`，不能通过目录推断。

## 路由契约

### 1. 输入契约

安装器只接受两类正式输入：

- `assistant_id`：由运行时识别阶段产出。
- `scope`：由安装命令或调用方明确提供，表示本次安装属于 `project` 还是 `global`。

项目目录、已有技能目录、历史安装结果都不属于正式输入。

### 2. 统一标识

运行时识别阶段最终只能产出统一的逻辑标识，例如：

- `claude`
- `qoder`
- `lingma`
- `trae`
- `generic`

后续映射、日志、测试、错误处理全部围绕这个统一标识展开，而不是直接围绕某个特定环境变量名、某个进程名或某种目录结构展开。

### 3. 统一注册表

安装器必须通过单一注册表解析目标路径，逻辑形式如下：

```text
TARGETS[assistant_id][scope]
```

对于项目级安装，映射固定为：

- `claude -> .claude/skills`
- `qoder -> .qoder/skills`
- `lingma -> .lingma/skills`
- `trae -> .trae/skills`
- `generic -> .agents/skills`

对于全局安装，注册表也必须继续使用同一个 `assistant_id -> scope -> target` 结构；如果某个助手没有合法的全局目标定义，必须显式报错，不允许退回猜测逻辑。

## 运行时识别策略

### 推荐的分层策略

考虑到现实里并不是所有助手今天都提供统一、稳定、显式的身份变量，识别阶段采用分层方案：

1. 优先读取宿主直接暴露的显式身份信号。
2. 如果没有显式信号，再根据宿主进程、启动器、可执行路径、宿主元数据等运行环境特征识别。
3. 将识别结果规范化成统一的 `assistant_id`。
4. 如果缺失、冲突或无法唯一识别，则失败。

一旦上层已经给出明确身份，下层只能用于校验或补充诊断，不能改写结果；如果上层与下层结论不一致，必须按冲突处理而不是择一继续。

### 允许的证据来源

可接受的证据来自宿主或运行时环境，例如：

- 宿主显式注入的环境变量
- 宿主提供的进程标识、应用标识或启动器标识
- 宿主暴露的运行时元数据
- 安装器自身已知且可验证的助手宿主特征

这些信号的共同要求是：它们描述的是“当前是谁在运行安装器”。

### 禁止的证据来源

以下信息一律不得参与助手身份判定：

- 项目根目录下是否存在 `.qoder/`
- 项目根目录下是否存在 `.trae/`
- 项目根目录下是否存在 `.claude/`
- 历史安装残留目录
- 任何基于仓库文件结构的存在性猜测

原因很简单：这些信息描述的是仓库历史，不是当前执行者。

## 决策顺序与失败策略

安装器的决策顺序必须固定，不能掺入回退猜测：

1. 收集运行时环境信号。
2. 识别并规范化 `assistant_id`。
3. 校验 `assistant_id` 是否唯一且受支持。
4. 校验 `scope` 是否合法。
5. 查询 `TARGETS[assistant_id][scope]`。
6. 输出解析结果日志，例如 `assistant=trae, scope=project, target=.trae/skills`。
7. 执行安装。

以下情况必须直接失败：

- 没有任何有效的宿主信号。
- 宿主显式信号与宿主特征识别结果互相冲突。
- 多个信号指向不同助手，无法唯一判定。
- 识别出的 `assistant_id` 不在受支持集合中。
- `scope` 非法。
- `TARGETS[assistant_id][scope]` 未定义。

失败时必须输出足够的诊断信息，至少包括：

- 读到了哪些信号
- 每个信号归一化后对应什么助手
- 为什么结果被判定为缺失、冲突或未知
- 因此未执行安装

## 兼容性与扩展性

### 统一协议，不做每助手专线逻辑

安装器不应为某个助手单独写一套“如果是 Trae 就这样，否则如果是 Qoder 就那样”的分叉流程。助手差异只允许出现在两处：

- 运行时识别规则中，负责把宿主特征归一化成统一 `assistant_id`
- 目标注册表中，负责声明该 `assistant_id` 在不同 `scope` 下的目标路径

除此之外，安装主流程应保持一致。

### 新增助手的准入方式

新增一个助手时，只需要补两类信息：

- 如何从它的宿主环境识别出稳定身份
- 它在注册表中的目标路径定义

不应要求重写安装决策树，也不应增加新的目录推断分支。

### 面向未来的演进方向

长期理想形态是所有助手宿主都能显式提供标准化身份信号。届时，分层识别中的“宿主特征推断”可以逐步退化为兼容层，但统一 `assistant_id -> scope -> target` 契约不变。

## 可信边界

这套设计的可信边界非常明确：

- 可以信任宿主或运行时环境信号。
- 不可以信任项目目录结构来表达当前助手身份。

因此正确性目标不是“任何情况下都想办法装进去”，而是：

- 能唯一判定时，装到正确位置。
- 不能唯一判定时，拒绝安装，避免误装到错误目录。

这比“尽量猜一个目录继续安装”更安全，因为误装会污染项目并制造后续排查成本。

## 验证要求

后续实现至少需要覆盖以下验证层面。

### 单元级验证

- 不同宿主信号能否被正确归一化到统一 `assistant_id`
- 冲突信号是否会被识别为失败
- 未知信号是否会被识别为失败
- `TARGETS[assistant_id][scope]` 是否返回正确目标
- 缺失映射时是否显式失败

### 集成级验证

- 当前运行环境是 Trae，且项目里同时存在 `.qoder/` 时，目标仍为 `.trae/skills`
- 当前运行环境是 Qoder，且项目里同时存在 `.trae/` 时，目标仍为 `.qoder/skills`
- 当前运行环境是 Claude，且项目里已有其他助手目录时，目标仍为 `.claude/skills`
- 未识别运行环境且项目里同时存在多个助手目录时，安装必须失败且不能猜测
- 同一助手在 `project` 与 `global` 两种 scope 下都通过同一注册表解析

### 诊断验证

- 成功路径会打印解析结果日志
- 失败路径会打印冲突或缺失原因
- 诊断中可以看出最终为什么没有执行安装

## 验收示例

以下场景必须成立，才能认为这套路由策略正确落地：

- `Trae runtime + existing .qoder/ -> install to .trae/skills`
- `Qoder runtime + existing .trae/ -> install to .qoder/skills`
- `unrecognized runtime + multiple assistant directories -> fail without guessing`

这三类示例分别锁定了：

- 运行时信号优先于项目目录
- 设计适用于所有助手，而不是 Trae 特例
- 失败策略以“拒绝误装”为目标，而不是继续猜

## 实施约束

实现时必须保持以下硬约束：

- 不允许保留目录存在性回退逻辑作为“兼容模式”
- 不允许把项目扫描结果混入 `assistant_id` 判定
- 不允许在未知助手上默认落到某个通用项目目录
- 不允许因为仓库里已有某个助手目录就覆盖运行时识别结果

只要这些约束被破坏，原问题就会以另一种形式重新出现。
