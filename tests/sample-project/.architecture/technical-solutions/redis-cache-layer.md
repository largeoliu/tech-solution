# Redis 缓存层方案

## 概述

在现有 API 层前面加一层 Redis 缓存，优化读取性能。缓存设置合理的过期策略，解决热点数据查询延迟问题。缓存作为读请求的第一层，未命中时回源到数据库。

## 缓存策略

### TTL 设计

| 数据类型 | TTL 配置 | 理由 |
|----------|----------|------|
| 用户配置 | 30 分钟 | 变更不频繁，允许一定延迟 |
| 热点商品 | 5 分钟 | 价格变动需要及时同步 |
| 枚举值 | 24 小时 | 几乎不变，可长时间缓存 |
| 会话信息 | 15 分钟 | 需要及时感知登录状态变更 |

### 缓存更新策略

- **Cache Aside**：业务代码主动更新缓存，读取时先查缓存，未命中回源
- **Write Through**：写操作同时写缓存和数据库，保证数据一致性
- **TTL 过期**：作为兜底策略，防止脏数据永久存在

### 过期策略原则

1. 热点数据设置较短 TTL，确保数据及时更新
2. 冷数据设置较长 TTL，减少回源压力
3. 使用随机偏移量避免缓存雪崩（同一时间大量 key 同时过期）
4. 重要数据配合主动删除策略，不完全依赖 TTL

## Key 规范

### 命名格式

```
cache:{module}:{entity}:{id}:{version}
```

### 组成部分

| 组成部分 | 说明 | 示例 |
|----------|------|------|
| `cache` | 固定前缀，标识缓存类型 | `cache` |
| `module` | 业务模块名 | `user`, `product`, `order` |
| `entity` | 实体类型 | `profile`, `detail`, `config` |
| `id` | 实体唯一标识 | `12345`, `uuid` |
| `version` | 数据版本 | `v1`, `v2` |

### 示例

- 用户 Profile：`cache:user:profile:12345:v1`
- 商品详情：`cache:product:detail:98765:v1`
- 订单配置：`cache:order:config:uuid-xxx:v1`

### 禁用规范

- 禁止在 key 中使用特殊字符（`:`、`/`、` `等除外）
- 禁止使用过长的 key（不超过 256 字节）
- 禁止使用可预测的 key 模式（防止业务冲突）

## 缓存架构图

```
                        读请求流程
                        ==========

    +--------+     +-------------+     +------------+     +----------+
    | Client | --> |  API Server | --> | Redis Cache| --> | Database|
    +--------+     +-------------+     +------------+     +----------+
                           |                   |
                           |                   +-- 命中 (Hit) --> 返回数据
                           |
                           +-- 未命中 (Miss) --> 回源查询 --> 写入缓存 --> 返回数据


                        缓存分层
                        ========

    +------------------------------------------+
    |              L1: 本地缓存                  |
    |         (Guava Cache / Caffeine)         |
    |         热点数据，TTL 30s ~ 1min          |
    +------------------------------------------+
                        |
                        v
    +------------------------------------------+
    |              L2: Redis 分布式缓存          |
    |         所有节点共享，TTL 5min ~ 24h      |
    +------------------------------------------+
                        |
                        v
    +------------------------------------------+
    |              L3: MySQL/PostgreSQL         |
    |                   持久化存储               |
    +------------------------------------------+
```

## 风险与应对

| 风险项 | 影响等级 | 缓解措施 |
|--------|----------|----------|
| 缓存穿透 | 高 | 空值缓存 + Bloom Filter 过滤不存在的数据 |
| 缓存雪崩 | 高 | TTL 添加随机偏移量（±10%），避免同时过期 |
| 缓存击穿 | 中 | 互斥锁保护热点 key，热点数据永不过期 |
| 数据不一致 | 中 | 短 TTL + 变更时主动删除缓存 |
| Redis 单点故障 | 高 | 主从复制 + 哨兵模式保证高可用 |
