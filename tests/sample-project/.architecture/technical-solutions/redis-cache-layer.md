# Redis 缓存层方案

## 概述

在现有 API 层前面加一层 Redis 缓存，优化读取性能。缓存设置合理的过期策略，解决热点数据查询延迟问题。缓存作为读请求的第一层，未命中时回源到数据库。

## 架构设计

### 缓存架构
```
[Client] -> [API Server] -> [Redis Cache] -> [Database]
                              |
                         未命中时回源
```

### 缓存策略
| 数据类型 | 过期策略 | 理由 |
|----------|----------|------|
| 用户配置 | 30min | 变更不频繁，允许一定延迟 |
| 热点商品 | 5min | 价格变动需要及时同步 |
| 枚举值 | 24h | 几乎不变，可长时间缓存 |
| 会话信息 | 15min | 需要及时感知登录状态变更 |

### 缓存 Key 规范
- 格式：`cache:{module}:{entity}:{id}:{version}`
- 示例：`cache:user:profile:12345:v1`

### 缓存更新策略
- **Cache Aside**：业务代码主动更新缓存
- **Write Through**：写操作同时写缓存和数据库
- **TTL 过期**：作为兜底策略，防止脏数据永久存在

## 实现方案

### 第一阶段：基础设施
1. 引入 Redis 客户端（Redisson）
2. 配置 Redis 连接池
3. 实现缓存客户端封装类

### 第二阶段：缓存接入
1. 用户Profile 缓存接入
2. 商品信息缓存接入
3. 枚举值缓存接入

### 第三阶段：监控告警
1. 缓存命中率监控
2. 慢查询告警
3. 缓存内存使用监控

### 任务分解
| 任务 | 负责人 | 依赖 |
|------|--------|------|
| Redis 环境准备 | backend-dev | 运维 |
| 缓存客户端封装 | backend-dev | 无 |
| 用户Profile缓存 | backend-dev | 封装类 |
| 商品缓存 | backend-dev | 封装类 |
| 监控告警 | backend-dev | 缓存接入 |

## 风险评估

| 风险项 | 影响 | 缓解措施 |
|--------|------|----------|
| 缓存穿透 | 高 | 空值缓存 + bloom filter |
| 缓存雪崩 | 高 | TTL 随机偏移量，避免同时过期 |
| 缓存击穿 | 中 | 互斥锁 / 热点数据永不过期 |
| 数据不一致 | 中 | 短 TTL + 变更时主动删除 |
