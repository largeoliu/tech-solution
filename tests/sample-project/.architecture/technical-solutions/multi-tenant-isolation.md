# 多租户数据隔离方案

## 概述

实现多租户数据隔离，采用行级租户标识方案。用户数据按租户隔离，现有 API 需要增加租户过滤。租户标识作为所有数据表的标准字段，通过统一中间件注入查询上下文。

## 架构设计

### 隔离方案选择：行级租户标识
**结论：采用行级租户标识（Tenant ID），理由如下：**
- 参照 repowiki 中现有租户隔离设计规范
- 相比独立数据库方案，运维成本更低
- 相比独立 Schema 方案，数据聚合查询更方便

**不可复用原因：**
- 现有系统无租户字段，无法通过简单改造实现
- 业务逻辑深度耦合租户概念，需重新设计数据模型

### 核心设计
- 所有业务表增加 tenant_id 字段（VARCHAR(64), NOT NULL, 默认值 'default'）
- 请求上下文存储当前租户 ID，通过 ThreadLocal 传递
- 数据访问层自动注入 tenant_id 过滤条件
- 租户隔离检查在 Service 层统一处理

### API 改造
| API 场景 | 改造方式 |
|----------|----------|
| 查询列表 | 自动追加 WHERE tenant_id = :current_tenant |
| 新增数据 | 自动填充 tenant_id 字段 |
| 跨租户查询 | 需显示声明 is_admin 权限 |
| 租户切换 | 通过 Header X-Tenant-ID 传递 |

## 实现方案

### 数据库改造
1. 所有业务表增加 tenant_id 字段
2. 建立 tenant_id 索引
3. 存量数据迁移到 'default' 租户

### 中间件层
1. 解析请求头 X-Tenant-ID，存入 ThreadLocal
2. 无效或缺失租户 ID 时返回 401
3. 超级管理员可指定任意租户

### 数据访问层
1. MyBatis 拦截器自动注入 tenant_id 条件
2. JPA @Where 注解声明租户过滤
3. 禁止手写不带 tenant_id 的查询 SQL

## 风险评估

| 风险项 | 影响 | 缓解措施 |
|--------|------|----------|
| 跨租户数据泄露 | 极高 | 数据访问层强制检查，代码审查重点 |
| 查询性能下降 | 中 | 复合索引 (tenant_id, business_key) |
| 存量数据迁移 | 高 | 提前备份，分批执行，监控回滚点 |
| 超级管理员权限滥用 | 高 | 操作审计日志，权限审批流程 |
