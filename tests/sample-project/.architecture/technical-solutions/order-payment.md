# 电商平台双通道支付模块方案

## 概述

为电商平台新增微信和支付宝双通道支付模块，支持退款、对账、异步回调，不影响现有订单流程。支付模块作为独立服务部署，通过消息队列与订单服务解耦。

## 架构设计

### 模块定位
支付模块为独立微服务，承载收银台、支付通道路由、退款处理、对账清算四大职责。

### 核心组件
- **Payment Gateway**：统一收银台，聚合微信/支付宝两大通道
- **Channel Router**：根据用户选择的通道路由到对应支付渠道
- **Refund Service**：处理退款请求，支持部分退款和全额退款
- **Reconciliation Service**：每日对账，生成差异报告

### 数据模型
| 实体 | 职责 | 关键字段 |
|------|------|----------|
| PaymentOrder | 支付订单 | order_id, channel, amount, status, created_at |
| RefundRecord | 退款记录 | payment_id, refund_amount, reason, status |
| ReconciliationLog | 对账日志 | date, channel, total_amount, diff_amount |

### 集成方式
- 与订单服务通过消息队列（Kafka）异步通信
- 支付结果通过异步回调通知订单服务状态变更
- 不直接依赖订单数据库，通过API查询订单信息

## 实现方案

### 阶段一：基础设施
1. 创建支付微服务模块，引入依赖：wxpay-sdk, alipay-sdk, kafka-client
2. 初始化支付数据库表结构
3. 配置微信/支付宝商户号和密钥

### 阶段二：核心支付流程
1. 实现统一收银台接口 POST /api/payment/create
2. 实现微信支付预下单接口（Native/API）
3. 实现支付宝预下单接口（当面付）
4. 实现支付结果回调通知接口
5. 实现支付状态查询接口

### 阶段三：退款与对账
1. 实现退款接口 POST /api/payment/refund
2. 实现对账文件拉取定时任务（T+1）
3. 实现对账差异检测和告警

### 兼容性设计
- 支付回调通过消息队列通知订单服务，订单服务无需同步等待
- 现有订单流程无需修改，通过事件驱动更新订单状态
- 支付服务独立部署，不影响主业务可用性

## 风险评估

| 风险项 | 影响 | 缓解措施 |
|--------|------|----------|
| 支付通道不可用 | 高 | 通道降级策略，支付宝/微信互为备份 |
| 重复支付 | 高 | 支付幂等性设计，订单号唯一约束 |
| 回调超时 | 中 | 回调重试机制 + 主动查询补偿 |
| 对账差异 | 中 | T+1差异告警，人工介入处理 |
| 资金安全 | 极高 | 签名验签、密钥隔离、审计日志 |
